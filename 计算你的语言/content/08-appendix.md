## A.2 每个活动的数学模型概要

以下用文字描述为主，辅以简洁的公式示意。所有公式只用于说明计算结构的逻辑，不需要数学背景也能理解方向。

### 阅读：贝叶斯更新

阅读的数学本质是 **贝叶斯推理**：

```
P(posterior | text) ∝ P(text | prior) × P(prior)
```

- **理解方向**：读者在看到文本之前，脑子里已经有了一个"先验认知"（P(prior)）。读的过程中，读者不断更新对文本的理解，同时用文本里的新信息校正自己的认知——最终得到一个更新后的"后验认知"（P(posterior)）。
- **阅读质量的本质**：不是记住了多少字，而是后验与先验之间的"信息距离"有多大。阅读如果没有改变先验，那就是"文字扫描"，不是真正的阅读。
- **计算视角的金句**：每次阅读都是一次贝叶斯更新——输入是文本，输出是更新后的世界模型参数。

### 思考：假设空间中的极值搜索

思考的数学本质是 **约束优化问题**：

```
argmax_{hypothesis ∈ S} f(hypothesis, constraints)
```

- **理解方向**：在一个定义好的假设空间 S 中，找到一个能最大化评估函数 f 的假设。S 是"所有可能的答案"——你的搜索范围有多大，决定了你的思考质量的上限。f 是"判断标准"——你用什么标准来判断一个答案好不好，决定了你最终选什么。
- **搜索策略决定效率**：广度优先（穷举）安全但慢，深度优先（一条路走到黑）快但容易错过更好解，启发式搜索（基于经验剪枝）是大多数高效思考者的秘密武器。
- **核心洞见**：大多数"想不清楚"的问题，根源要么是搜索空间画得太小（知道得太少），要么是评估函数设错了（拿错的尺子量对的路）。

### 表达：多维约束下的最优路径搜索

表达的数学本质是 **带约束的优化问题**：

```
min_{text_seq ∈ T} distance(intent_vector, decode(text_seq))
subject to: fact_constraint, emotion_constraint, relation_constraint, purpose_constraint
```

- **理解方向**：你的"想说的话"是一个意图向量。你要找到一段文本序列，使得这段文本被读者解码后的语义向量，和你脑子里的意图向量之间的距离最小——同时满足四个约束条件。
- **四维约束**：
  - 事实约束：信息不能歪曲
  - 情感约束：语气要传递正确情绪
  - 关系约束：编码方式要符合双方关系
  - 目的约束：输出后对方要知道该做什么
- **核心洞见**：90%的沟通问题出在"没考虑约束"而非"表达能力不足"。约束清单背下来，表达能力自动翻倍。

### 演讲：状态机路径规划

演讲的数学本质是 **有限状态机的路径规划**：

```
S₀ → S₁ → S₂ → ... → Sₙ
```

- **理解方向**：听众在听演讲的过程中，会经历一系列认知-情感联合状态的迁移。好演讲的本质不是"信息密度高"，而是"状态迁移效率高"——用最少的步骤、最短的时间，把听众从 S₀ 带到 Sₙ。
- **状态定义**：每个 Sₖ 是一个复合状态（认知状态 × 情感状态），例如：
  - S₀ =（"不关心这个话题" × "无聊"）
  - S₁ =（"好像有点意思" × "好奇"）
  - Sₙ =（"我要行动了" × "兴奋"）
- **核心洞见**：设计演讲时，先画状态迁移图（S₀→S₁→...→Sₙ），再为每个迁移点准备对应的"语言触发器"——这就是专业和不专业演讲者的分水岭。

### 辩论：对抗训练中的决策边界优化

辩论的数学本质是 **对抗训练（Adversarial Training）**：

```
decision_boundary = argmin_{B} error(positive_examples, negative_examples)
```

- **理解方向**：裁判（第三方）心里有一条决策边界——他认为"这个观点对""那个观点不对"的分界线。辩论双方轮流输入正例（支持自己观点的论据）和反例（反驳对方的论据），目的是让这条边界朝己方偏移。
- **关键变量**：
  - 正例质量：你的论据是否能有效拉边界
  - 反例质量：你是否能找到对方论据的"弱支撑"
  - 裁判先验：裁判一开始的偏见会影响边界初始位置和偏移难度
- **核心洞见**：辩论不是说服对方——对方的决策边界通常动不了。辩论是对裁判的训练。不明白这一点的人，是在跟空气打架。

### 说服：参数渐进更新（BP风格）

说服的数学本质是 **梯度下降式的参数更新**：

```
model_other(t+1) = update(model_other(t), training_data)
Δparams = -η · ∇loss(params, training_data)
```

- **理解方向**：对方的"世界模型"是一个复杂的参数系统。你说服他不是去"替换"这个系统——你是通过输入精心设计的"训练数据"，让他的模型自己触发参数更新。每一步只调一点（小学习率 η），让参数沿着损失函数下降的方向走。
- **最小修改量原则**：一次只改一个参数（一个信念维度），每次不超过5%。试图同时改多个参数——对方的模型会进入"保护模式"（抗拒），更新终止。
- **核心洞见**：最高级的说服不是让人接受你的结论——是让人接受你的坐标系。换坐标系的力量比调参数大一个数量级，但前提是你要先理解对方的坐标系是什么。

## A.3 核心概念定义

以下为全书中反复出现的核心概念，按字母顺序排列（中文为主，标注英文对应）：

### 世界模型（World Model）
每个人对世界的认知总集合——包括知识、信念、偏见、经验、价值观等。在本书中，世界模型被类比为AI模型中的参数系统：有新信息输入时，它会选择性更新部分参数。**阅读和说服的核心对象就是世界模型。**

### 语义空间（Semantic Space）
所有词语、概念按语义相似度映射到的高维坐标系。在语义空间中，意思相近的词（如"高兴"和"快乐"）距离近，意思相反的词（如"爱"和"恨"）距离远。**任何文字活动都是在这个空间中的向量操作。**

### 先验 / 后验（Prior / Posterior）
- **先验（Prior）**：在接触到新信息之前，主体已有的认知状态或信念分布。
- **后验（Posterior）**：接触并处理完新信息之后，更新后的认知状态。
- 在阅读和说服场景中最常用：先验→（输入新信息）→后验。更新的"量"是衡量阅读/说服效果的核心指标。

### 评估函数（Evaluation Function / Loss Function）
判断一个假设（方案、观点、答案）好不好的标准函数。思考的质量不取决于"想得努力不努力"，而取决于"你的评估函数有没有选对"。如果你的评估函数是"最快想到什么就用什么"，你的思考不可能好。

### 决策边界（Decision Boundary）
在辩论场景中，第三方（裁判/观众/读者）心里的一维度或多维度的"划线"——顺着这条线的一侧认为你对，另一侧认为你不对。辩论的目标不是消除这条线，是让它朝你的方向偏移。

### 约束空间（Constraint Space）
表达过程中必须满足的条件的集合。四个基本约束维度：事实（truthfulness）、情感（emotion）、关系（relationship）、目的（purpose）。约束空间越清晰，表达路径越明确。**表达式难在约束多，而不是表达能力差。**

### 状态迁移（State Transition）
演讲场景中，听众从一种认知-情感状态变为另一种状态的过程。每次状态迁移需要一个"语言触发器"——一段话、一个故事、一个数据、一个表情。好演讲的设计工作核心就是规划这些迁移点。

## A.4 速查决策树

在遇到一个实际沟通场景时，以下决策树帮你快速判断"我现在应该用哪个框架"：

```
面对一段文字/一个沟通场景
你处在什么状态？
│
├── 📖 在看（阅读别人写的东西）
│   → 使用【阅读框架】
│   ├── 检查：你是在做"文字扫描"还是"解压缩"？
│   ├── 问自己：我的先验更新了吗？
│   └── 工具：解压缩检查清单（三句真题）
│
├── 🤔 在想（面对一个问题需要判断）
│   → 使用【思考框架】
│   ├── 检查：搜索空间定义好了吗？边界在哪？
│   ├── 问自己：我的评估函数是什么标准？
│   └── 工具：搜索框架模板（定义空间→选策略→设评估函数→开搜）
│
├── ✍️ 在写（需要输出一段文字）
│   → 使用【表达框架】
│   ├── 检查：四维约束过了吗？（事实/情感/关系/目的）
│   ├── 问自己：读者解码后得到的是我的意图吗？
│   └── 工具：表达约束清单（写前30秒过四问）
│
├── 🎤 在讲（需要对一群人说话）
│   → 使用【演讲框架】
│   ├── 检查：听众的初始状态分析了吗？
│   ├── 问自己：我的状态迁移图画了吗？
│   └── 工具：状态迁移设计法（S₀→S₁→...→Sₙ）
│
├── ⚔️ 在吵（有争议，需要第三方判断）
│   → 使用【辩论框架】
│   ├── 检查：裁判是谁？他的初始边界在哪？
│   ├── 问自己：我是在对裁判说话还是在对对手说话？
│   └── 工具：边界偏移分析法（正例拉/反例推/裁判先验）
│
└── 🫂 在劝（需要改变另一个人的想法）
    → 使用【说服框架】
    ├── 检查：对方的先验模型长什么样？
    ├── 问自己：我这次只调一个参数吗？步长够小吗？
    └── 工具：模型分析清单（诊断prior→设计训练数据→控制步长→换坐标系）
```
