# 标准作业书 - 代码风格标准 v3.0

> 版本 v3.0 · 生效日期：____ 年 __ 月 __ 日
>
> ——— 让你写的代码，别人拿起来就能干活。

---

团队效率的真正瓶颈不是"缩进用空格还是 Tab"这种格式化工具就能解决的事。
真正的瓶颈是——拿到一段代码，你得花一半时间猜"原作者到底想干嘛"。

所以 v3.0 的核心升级是：**从顶层结构到底层细节，全链路降低认知负载。**

四层约束，一套搞定：

> **目录标准** → 一眼看懂项目骨架
> **文件标准** → 一眼看懂文件职责
> **函数标准** → 一眼看懂函数意图
> **工作流标准** → 一眼看懂开发流程

格式规范还在，但退居细则层——它是工具的事，不是思想的事。

---

# 一、总纲 · 核心原则

> 把业界最优秀的思想吸收进来，统一为一套标准。

## 1.1 第一原则：看名字就懂

代码首先是写给**未来的人（包括未来的自己）**看的。

- **函数名、变量名、类型、接口** → 看到名字直接知道目的
- **禁止万能动词**：不写 `handle()`、`process()`、`execute()`、`run()`，写 `createMemory()`、`retrieveUser()`、`deleteOrder()`
- **让代码自己表达意图**，减少读者的推理成本

### ❌ 坏示例 —— 猜谜游戏

```python
# 这俩函数到底干嘛的？得点进去看。
def run():
    ...

def process(data):
    ...

result = run()
```

### ✅ 好示例 —— 一目了然

```python
def create_memory_index(documents: list[str]) -> IndexResult:
    """将文档写入向量索引。"""

def retrieve_user_by_email(email: str) -> User | None:
    """根据邮箱查找用户。"""

index = create_memory_index(docs)
user = retrieve_user_by_email("user@example.com")
```

**关键**：看到名字，不用点开函数体，你就知道它在干嘛。

---

## 1.2 第二原则：看流程就懂

减少不必要的抽象层级。

- **目录结构一眼看出项目怎么运转**
- **不叠十层** Controller / Service / Manager / Factory / Builder / Strategy / Adapter
- **追求"看到文件直接知道执行过程"**，减少跳转成本

### ❌ 坏示例 —— 十层大礼包

```
src/
├── controllers/
│   └── user_controller.py
├── services/
│   └── user_service.py
├── managers/
│   └── user_manager.py
├── factories/
│   └── user_factory.py
├── builders/
│   └── user_builder.py
├── repositories/
│   └── user_repo.py
```

看了这个目录，你知道项目怎么跑的吗？不知道。

### ✅ 好示例 —— 直来直往

```
src/
├── main.py            # 入口：启动服务器
├── core/              # 核心业务逻辑
│   ├── user.py        # 用户相关
│   └── order.py       # 订单相关
├── tools/             # 工具函数
├── config/            # 配置
└── models/            # 数据模型
```

看了这个目录，你能猜出项目大概怎么跑。

---

## 1.3 设计哲学

> **规则一：代码首先是写给未来的人看的。**
> **规则二：让意图自解释，让流程一目了然。**
> **规则三：同一种事情只有一种写法——LLM 和人都受益。**

这三点是灵魂。后面的四层标准、格式细则、工具链，全是为了落地这三点。

---

## 1.4 新旧标准兼容原则

- **新项目 → 必须用 v3.0 新标准**
- **老项目 → 按需迁移，不强求**
- **涉及大重构或新模块时 → 优先按新标准写**

别把老代码全翻一遍，先把手上的新代码写好。

## 2.2 第二层 · 文件标准

### 文件名

- 全项目统一使用 **kebab-case**（小写+连字符）
- React 组件用 **PascalCase**（因为框架社区惯例就这样）

```text
# ✅ 好
user-service.ts
memory-index.ts
temperature-sensor.py
UserOrderList.tsx     # React 组件
test_memory.py

# ❌ 不好
UserService.ts        # 文件名不要驼峰
utils.ts              # 太泛，"工具"到底是什么工具？
handleEverything.py   # 一个文件干所有事
```

### 文件职责

- **一个文件只做一件事**
- 有明确的一句话职责描述
- 如果文件职责需要两句话，拆文件

### 测试文件

- Python：`test_xxx.py`
- JS/TS：`xxx.test.ts` / `xxx.spec.ts`

## 2.3 第三层 · 函数标准

### 函数命名 —— 核心原则落地

> 这层的核心意义是：**看到名字就懂。**

| ✅ 好的命名 | ❌ 不好的命名 | 为什么 |
| --- | --- | --- |
| `create_memory()` | `handle_memory()` | "处理记忆"到底干嘛？ |
| `retrieve_user_by_email()` | `get_user()` | 按什么拿用户？ |
| `delete_order()` | `process_order()` | 处理订单是取消还是发货？ |
| `calculate_discount()` | `run()` | 跑什么？ |
| `send_email()` | `execute()` | 执行什么？ |

### 函数职责

- **一个函数只做一件事**（单一职责）
- **不超过 50 行**（太长了就要拆）
- **布尔参数尽量拆成两个函数**

```python
# ❌ 不好 —— bool参数埋雷
def send_email(recipient: str, body: str, is_batch: bool):
    if is_batch:
        ...
    else:
        ...

# ✅ 好 —— 拆成两个函数
def send_email(recipient: str, body: str):
    """发送单封邮件。"""

def send_batch_email(recipients: list[str], body: str):
    """批量发送邮件。"""
```

### 参数命名

参数名也要自解释。避免歧义、避免单字母。

```python
# ❌ 不好
def run(a, b):
    ...

# ✅ 好
def create_memory_index(documents: list[str], options: IndexOptions) -> IndexResult:
    ...
```

## 2.4 第四层 · 工作流标准

> 每个需求/任务的标准作业流程（TPS 标准作业书思想）。

### 开发流程

```text
需求分析 → 规格定义 → 实现编码 → 代码审查 → 测试验证 → 发布上线
```

每步交付物：

| 阶段 | 交付物 | 标准 |
| --- | --- | --- |
| 需求分析 | 需求文档 | 明确做什么、为什么做 |
| 规格定义 | 接口设计 / 技术方案 | 明确怎么做 |
| 实现编码 | 代码 | 符合本标准的代码 |
| 代码审查 | 审查记录 | 验收清单打勾 |
| 测试验证 | 测试通过报告 | 单元测试+集成测试 |
| 发布上线 | 上线记录 | 回滚方案有备无患 |

### Git 分支规范

```
main ← develop ← feature/xxx-task
```

- **main**：生产分支，只接收经过测试+审查的合并
- **develop**：开发分支，日常开发基线
- **feature/xxx-task**：功能分支，每个任务一条分支

### Commit 规范

沿用通用格式（不用重新发明一遍）：

```
<type>(<scope>): <subject>
```

| type | 含义 |
| --- | --- |
| `feat` | 新功能 |
| `fix` | 修复bug |
| `docs` | 文档 |
| `style` | 代码格式 |
| `refactor` | 重构 |
| `perf` | 性能优化 |
| `test` | 测试 |
| `chore` | 杂项 |
| `ci` | CI/CD |

```bash
# ✅ 好
feat(订单模块): 新增批量导出功能
fix(用户中心): 修复手机号格式校验不通过
refactor(支付模块): 抽离支付策略

# ❌ 不好
update code        # 没说是什么
fix bug            # 太模糊
```

# 三、格式细则

> 格式规范有意义，但它是工具的事——Prettier、Black 一把梭就搞定，不需要人死记硬背。
>
> 这里精简保留核心规则，供手动审查时参考。

---

## 3.1 通用格式

| 项目 | 标准 | 说明 |
| --- | --- | --- |
| 缩进 | 4空格（JS/TS用2空格） | 各语言按社区惯例 |
| 行宽 | 100字符 | 不超100，工具自动格式化 |
| 编码 | UTF-8 | 统一编码 |
| 文件末尾 | 保留一个空行 | POSIX标准 |
| 行尾空格 | 一律删除 | diff干净 |

## 3.2 语言级命名规则

| 场景 | Python | JS/TS |
| --- | --- | --- |
| 模块/文件 | snake_case | kebab-case |
| 类 | PascalCase | PascalCase |
| 函数 | snake_case | camelCase |
| 变量 | snake_case | camelCase |
| 常量 | UPPER_SNAKE | UPPER_SNAKE（or camelCase） |
| 接口（TS） | — | PascalCase |
| React 组件 | — | PascalCase+`.tsx` |

## 3.3 布尔变量命名

- 统一前缀：`is_` / `has_` / `should_` / `can_`
- 禁止否定式布尔名

```python
# ❌ 不好
is_not_disabled = True
no_errors = True

# ✅ 好
is_enabled = True
has_error = False
```

## 3.4 注释规范

- **所有公开 API / 函数必须有文档注释**
- **注释只写"为什么"，不写"是什么"** —— 代码自己说明"是什么"
- **TODO 必须标注责任人**

```python
# ✅ 好注释 —— 写为什么
def calculate_discount(price: float, user_level: int) -> float:
    """根据用户等级计算折扣金额。

    策略：等级越高折扣越低。
    原因是高级用户已经是忠实用户，不需要大折扣吸引，
    折扣资源应该用在拉新上。
    """

# ❌ 坏注释 —— 写是什么（废话）
# 给x加1
x = x + 1

# ✅ TODO
# TODO(@责任人): 列表为空时会崩溃，需要处理边界情况

# ❌ TODO不留责任人的
# TODO: 后面再优化
```

## 3.5 异常处理原则

- **永不吞异常**：空的 `except` / `catch` 是代码犯罪
- **不要用异常控制正常流程**
- **异常要带上下文**

```python
# ❌ 不好 —— 吞异常
try:
    result = api.call()
except:
    pass

# ✅ 好
try:
    result = api.call()
except ConnectionError:
    log.warning("API连接失败，跳过")
    result = None
```

## 3.6 导入规范

按以下顺序分组，每组之间**空一行**：

1. 标准库
2. 第三方库
3. 本地库

```python
# ✅ 好
import os
import sys

import requests
import numpy as np

from myproject.models import User
```

## 3.7 代码红线（八条一条不少）

> 触之即死，代码审核看到必须打回。

| # | 红线 | 正确做法 | 说明 |
| --- | --- | --- | --- |
| 1 | 硬编码敏感信息 | 环境变量/配置文件 | 密码、密钥、Token、API Key |
| 2 | 硬编码魔数 | 命名常量 | 比如 3600 → `MAX_TIMEOUT_SECONDS` |
| 3 | `print` / `console.log` 用于生产日志 | logging 框架 | 日志要分级、可配置 |
| 4 | 超过 **100行** 的函数 | 拆成小函数 | 一个函数只做一件事 |
| 5 | 超过 **500行** 的文件 | 拆成多个文件 | 单一职责原则 |
| 6 | 空的 `except` / `catch` | 要么处理要么解释 | 吞异常 = 留坑 |
| 7 | 直接拼接 SQL | 参数化查询 | SQL注入警告 ⚠️ |
| 8 | `eval()` / `exec()` 执行用户输入 | 安全的解析器 | 代码注入警告 ⚠️ |

### 魔数示例

```python
# ❌ 不好 —— 3600是什么？
if total > 3600:
    print("超时")

# ✅ 好
MAX_TIMEOUT_SECONDS = 3600
if total > MAX_TIMEOUT_SECONDS:
    print("超时")
```

### SQL 示例

```python
# ❌ 不好 —— 拼接SQL
query = f"SELECT * FROM users WHERE name = '{user_input}'"

# ✅ 好
query = "SELECT * FROM users WHERE name = ?"
cursor.execute(query, (user_input,))
```

# 五、验收清单

> 合并前对着打勾，缺一个都不许合。

- [ ] 目录结构符合项目类型骨架（Python / JS-TS / 嵌入式）
