# 第五章 Cheat Engine 内存搜索

> **本章内容**：Cheat Engine（CE）是游戏逆向最常用的内存搜索工具。本章涵盖 CE 的核心概念和 Tutorial（教程）全部 9 步的知识点——尽管 ARM64 Win11 上 CE 的驱动拦截功能部分受限，但原理通用。

---

## 5.1 CE 是什么

**Cheat Engine（CE）** 是一个开源的内存扫描器/调试器。它让你能**搜索、查看、修改**一个进程在内存中的数值。

### 它能做什么

| 应用 | 例子 |
|:----|:-----|
| 改单机游戏 | 血量锁满、金币改 99999 |
| 分析游戏逻辑 | 找出血量、坐标、技能冷却的地址 |
| 制作外挂 | 配合代码注入实现自动回血/无限子弹 |
| 逆向分析 | 在找不到源码时，定位关键数据/代码 |

### 核心思想

> **游戏里所有的值——血量、金币、坐标、子弹数——都存储在内存的某个地址上。CE 的工作就是找到那个地址，然后改它。**

---

## 5.2 CE 界面一览

- **进程选择窗口**：附加到目标进程（如游戏）
- **Value / Scan Type / Value Type**：设置搜索参数
- **First Scan / Next Scan**：首次搜索 / 再次过滤
- **地址列表**：找到的地址可以双击锁定在这里，批量修改/冻结
- **内存视图**：以十六进制查看/编辑内存

## 5.3 Tutorial 九步核心知识点

> CE 自带的 Tutorial（教程程序）是一个训练关卡，逐步教会你所有核心技巧。

---

### Step 1: 进程附加

| 概念 | 说明 |
|:----|:------|
| **附加（Attach）** | CE 和目标进程建立连接，才能读写它的内存 |
| **进程列表** | 所有正在运行的程序都在这里，选教程进程 |

---

### Step 2: 精确数值扫描（Exact Value Scan）

**场景**：你知道当前数值（如血量 100），它变化后也看得到。

**操作**：
1. Value 输入当前值 → Scan Type 选 **Exact Value**
2. 点 **First Scan** → 列出所有匹配地址
3. 改变数值 → 输入新值 → 点 **Next Scan**
4. 重复直到只剩 **唯一地址**
5. 双击加入列表，改值

**核心口诀**：看数字→搜数字→变数字→再搜数字→锁定唯一地址→改它！

**Step 2 密码：090453**

---

### Step 3: 未知初始值扫描（Unknown Initial Value）

**场景**：数值在变，但你**看不到具体数值**（如只有血条没有数字）。

**操作**：
1. Scan Type 选 **Unknown Initial Value** → 点 **First Scan**
2. 让数值变化（掉血）
3. Scan Type 改 **Decreased Value（减少的数值）** → **Next Scan**
4. 重复变化+过滤，直到只剩少数地址
5. 选中看起来合理的值，加入列表，改值

**关键**：你不是在搜一个精确数字，而是在过滤掉"没变的地址"。

**Step 3 密码：419482**

---

### Step 4: 浮点数扫描（Floating Points）

**场景**：数值带小数（如血量 100.0、弹药 100.0）。

**知识点**：
| 类型 | 位宽 | 何时使用 |
|:----|:-----|:---------|
| **Float**（单精度浮点数） | 4字节 | 血量、坐标、角度等 |
| **Double**（双精度浮点数） | 8字节 | 精度要求更高的小数 |

**操作**：和 Exact Value 一样搜，但 Value Type 选 **Float** 或 **Double**。

**Step 4 密码：890124**

---

### Step 5: 找出谁在改写地址（Find out what writes）

**场景**：数值地址每次重启游戏会变，需要找到修改这个地址的**代码**。

**操作流程**：
1. 搜到地址 → 加入列表
2. **右键地址 → Find out what writes to this address**
3. 弹窗打开后，到目标程序触发数值变化
4. 弹窗捕获到一条汇编代码（如 `mov [eax], 00000001`）
5. 选中代码 → **Replace（替换成 nop，即空指令）**
6. 值被冻住，不再变化

**原理**：在内存中设置"访问断点"，当目标地址被写入时拦截写入指令。

**⚠️ ARM64 Win11 踩坑**：CE 的驱动拦截功能在 ARM64 Win11 上可能无法正常工作，捕获不到代码。但这不影响原理学习。

**Step 5 密码：——（需实测）**

### Step 6: 指针（Pointers）

**场景**：数值的地址每次重启都会变，但有一个**指针**的地址是固定的。

**定义**：

> **指针 = 一个存着"地址"的变量。**
>
> 指针地址固定不变，但指针里的内容（目标地址）可以变。

**单级指针示例**：
```
固定指针地址 [0x004000A0] = 0x12345678  ← 指向血量地址
     ↓
血量地址     [0x12345678] = 100         ← 实际血量值
```

**操作流程**：
1. 搜到血量地址 → Find out what accesses
2. 捕获到汇编代码如 `mov [esi+12], eax`
3. **esi** ＝ 指针，**12**（十六进制）＝ 偏移量
4. 添加手动地址 → 勾 **Pointer** → 填指针地址 + 偏移量
5. 改值并冻结 → 确认指针工作正常

**核心思想**：程序通过"基址 + 偏移"来定位数据，你只需要保存基址和偏移量，游戏重启也能找到。

### Step 7: 代码注入（Code Injection）

**场景**：不满足于只改数值，你想**修改游戏的行为逻辑**。比如把"扣血按钮"改成"加血按钮"。

**原理**：

```
游戏原来指令：
  sub [health], 1   ← 扣血1点

你注入的指令：
  add [health], 2   ← 加血2点

注入后（流程被改写）：
  jmp 你代码地址 → [add [health], 2] → 跳回原流程
```

**操作流程**：
1. 搜到血量地址 → Find out what writes
2. 捕获减血代码 → 记下地址
3. **Ctrl+A（打开 Auto Assembler）**
4. **Template → Code Injection** → 填入代码地址
5. 在 `newmem:` 下面写：
   ```asm
   add [血量地址], 2
   ```
6. 点 **Execute** → 按钮从扣血变回血

**关键术语**：
| 术语 | 说明 |
|:----|:------|
| **Code Cave（代码洞穴）** | 注入代码占用的内存空间 |
| **originalcode** | 游戏原本的指令 |
| **newmem** | 你写的自定义代码 |
| **nop** | 空指令（什么也不做） |

**现实应用**：自动回血、无限弹药、一击必杀等外挂功能的核心实现手法。

**Step 7 密码：013370**

---

### Step 8: 多级指针（Multilevel Pointers）

**场景**：血量地址不是直接通过一个指针找的，而是**指针指向指针，指针又指向指针……**最终才指向血量。

**多级指针链示例**：
```
client.dll + 0xDEA8C  ← 基址（固定不变，绿色）
    ↓ [+0x0]
指针1（动态地址）
    ↓ [+0x10]
指针2（动态地址）
    ↓ [+0x4]
指针3（动态地址）
    ↓ [+0x8]
血量地址 → 血量值
```

**操作思路（递归）**：
1. 找血量地址 → 看谁访问它 → 得到指针1 + 偏移1
2. 找指针1的地址 → 看谁访问它 → 得到指针2 + 偏移2
3. 找指针2的地址 → 看谁访问它 → 得到指针3 + 偏移3
4. 重复直到找到**绿色（静态）地址**

**颜色含义**：
| 地址颜色 | 含义 |
|:---------|:-----|
| **绿色** | 静态地址（基址，永远不变） |
| **蓝色/黑色** | 动态地址（每次启动会变） |

**现实意义**：外挂保存的不是固定地址，而是**基址 + 偏移链**。每次启动游戏走一遍指针链，总能找到血量。

**Step 8 密码：525927**

---

### Step 9: 区分敌我（Shared Code）

**场景**：你要改血量的代码，但你和敌人用的是同一段代码。如果你把减血禁掉，敌人也不减血了。

**核心问题**：怎么让代码只对自己生效，不对敌人生效？

**解决思路**：
1. 找到减血的代码
2. 对比己方单位和敌方单位的**数据结构差异**
3. 找到一个区分标志（如队伍ID、对象类型、玩家名指针）
4. 注入带判断条件的代码：

```asm
// 伪代码逻辑
if (对象.队伍ID == 己方) {
    // 跳过减血 → 自己不掉血
} else {
    // 正常减血 → 敌人照常掉血
}
```

**常见区分标志**：
| 标志类型 | 说明 |
|:---------|:------|
| 队伍ID | 不同队伍用一个数字标记（如 0=己方, 1=敌方） |
| 对象类型 | 玩家/怪物/NPC 各用不同ID |
| 玩家名指针 | 指向玩家名字的地址 |
| 函数指针表 | 每个对象的第一个字段通常是函数表指针 |

**现实应用**：
| 场景 | 做法 |
|:----|:------|
| 自己无敌 | 判断是自己 → 跳过减血 |
| 秒杀敌人 | 判断是敌人 → 减一个巨大数值 |
| 队友不死 | 判断是己方队伍 → 不减血 |
| 只改自己弹药 | 判断是己方角色 → 不减弹药 |

**Step 9 密码：31337157**

## 5.4 CE 核心概念总结

| 概念 | 一句话 |
|:----|:-------|
| **Exact Value** | 知道数值，直接搜 |
| **Unknown Initial Value** | 看不到数值，靠变化过滤 |
| **Float/Double** | 小数也得搜，选对类型 |
| **Find out what writes** | 抓谁在改这个地址 |
| **Pointer** | 地址会变，用固定指针找 |
| **Multilevel Pointer** | 指针套指针，递归找到基址 |
| **Code Injection** | 替换游戏代码，改行为 |
| **Shared Code** | 敌我共用代码，加条件区分 |

---

## 5.5 ARM64 Win11 踩坑记录

| 问题 | 原因 | 建议 |
|:----|:-----|:-----|
| x64dbg 不兼容 | ARM64 不支持运行 x86 调试器 | 用 WinDbg 替代 |
| CE Find out what writes 无法捕获代码 | CE 驱动在 ARM64 上可能有兼容问题 | 原理照学，实操等 x86 机器 |
| PE-bear 界面异常 | ARM64 新版 GUI 兼容问题 | 用 CFF Explorer 替代 |
| IDA Free 不支持 ARM64 | 免费版只支持 x86/x64 | 用 Ghidra 替代 |

## 本章小结

1. **CE 本质**：搜索进程内存，定位并修改数据
2. **精确搜索**：知道数值直接搜（Step 2）
3. **模糊搜索**：不知道数值靠变化过滤（Step 3）
4. **指针**：地址会变时，用固定基址+偏移量定位（Step 6/8）
5. **代码注入**：替换游戏指令来改变游戏行为（Step 7）
6. **区分敌我**：在共享代码中加条件判断（Step 9）
7. **9 步下来**：你就从只会搜血量的菜鸟，变成懂指针链、代码注入、条件过滤的逆向入门者

**一句话：CE 是游戏逆向的利器——搜数字找地址，改数字改结果，注代码改逻辑。**
