## 第三章：反调试技术

### 3.1 反调试的本质

**一句话：软件想办法检测自己是不是在被调试，如果是，就改变行为。**

```
正常用户双击 → 程序正常运行

逆向者用调试器打开 → 程序发现有人盯着它
                     → 假装正常 / 直接崩溃 / 进入错误的路径
                     → 逆向者一脸懵
```

> **🖥️ 类比：** 反调试就像设备上的 **安全光幕**——有人把手伸进去（调试器），机器立刻停机，不让你看到正常运行时的动作。

### 3.2 反调试的"三板斧"

所有反调试技术归结起来就三种思路：

| 思路 | 比喻 | 举例 |
|:-----|:-----|:------|
| **① 检测调试器** | 装个摄像头，看到调试器就报警 | `IsDebuggerPresent()`、`NtGlobalFlag` |
| **② 干扰调试器** | 故意制造混乱，让调试器看不清楚 | 代码自修改、垃圾指令插入 |
| **③ 时间炸弹** | 检测执行时间，发现太慢就异常 | `rdtsc` 时间差检测 |

### 3.3 五种常见反调试手段

#### 手段①：IsDebuggerPresent() — 最简单也最经典

```c
// Windows API：检查自己是否在被调试
BOOL IsDebuggerPresent(VOID);
```

**原理**：读取 PEB（进程环境块）的 `BeingDebugged` 标志位。

**反汇编代码**：
```asm
CALL DWORD PTR [kernel32!IsDebuggerPresent]  ; 调用检查函数
TEST EAX, EAX                                  ; 检查返回值是否为0
JE   normal_flow                               ; =0（没被调试），正常走
JMP  crash_or_exit                             ; ≠0（被调试了），退出
```

**如何绕过？**

```
方法1：改跳转
  找到 TEST EAX, EAX → JE normal_flow
  把 JE → JMP（永远正常走）

方法2：改返回值
  在 CALL 返回后，手动把 EAX 改成 0

方法3：用 ScyllaHide 插件
  x64dbg 插件，自动 Hook 这个 API，永远返回 FALSE
```

#### 手段②：NtGlobalFlag — PEB 中的隐藏痕迹

调试器加载程序时，会在 PEB 的 `NtGlobalFlag` 字段留下痕迹：

```c
// PEB 结构体
typedef struct _PEB {
    BOOLEAN BeingDebugged;      // → IsDebuggerPresent() 读的就是这个
    UCHAR NtGlobalFlag;         // ← 调试器加载时这个字段被改
    ...
}
```

**正常情况**：`NtGlobalFlag = 0`
**被调试**：`NtGlobalFlag & 0x70 = 0x70`

**反汇编代码**：
```asm
MOV EAX, DWORD PTR FS:[0x30]   ; 获取 PEB 地址
MOV AL, BYTE PTR [EAX+0x68]    ; 读取 PEB+0x68 = NtGlobalFlag
AND AL, 0x70                    ; 检查调试器痕迹
JNE 检测到调试器                 ; 有标记，跳走
```

**绕过方法**：把 `MOV AL, [EAX+0x68]` 改成 `MOV AL, 0`（硬改），或用 ScyllaHide。

#### 手段③：rdtsc 时间差检测（最难绕过的之一）

**不依赖任何 API**，直接读 CPU 内部计数。

```asm
RDTSC                      ; 读取 CPU 时钟周期数 → 时间点 T1
MOV  [T1], EAX             ; 存下来
CALL 某函数                 ; 执行一段代码
RDTSC                      ; 读取 → 时间点 T2
MOV  [T2], EAX
SUB  EAX, [T1]             ; T2 - T1 = 耗时
CMP  EAX, 100000           ; 超过阈值？
JA   检测到调试器            ; 太慢了 → 被单步跟踪
```

**原理**：
- 正常执行：一段代码几十个指令，CPU 几微秒就跑完了
- 调试器单步：每步都要停 → 慢几百倍

**绕过方法**：
```
1. 在第一个 RDTSC 后面下断点，记下 EAX 值
2. 走到第二个 RDTSC，手动改 EAX
3. 让差值看起来在正常范围内
```

#### 手段④：TLS 回调函数（最阴险，入口点之前执行）

```c
// TLS = Thread Local Storage
// 正常程序：入口点是第一条指令
// 带 TLS 反调试的程序：
//   1️⃣ 操作系统先加载 TLS 回调函数
//   2️⃣ TLS 回调执行反调试检测
//   3️⃣ 检测到调试 → 直接退出
//   4️⃣ 入口点根本没机会执行！
```

**绕过方法**：
```
在 x64dbg 中：
  选项 → 设置 → 事件 → 勾选"在 TLS 回调时中断"
这样 TLS 回调的第一条指令就会被断下来。
```

#### 手段⑤：代码自修改（Self-Modifying Code）

代码在运行中修改自己，让静态分析和断点都失效。

```asm
; 文件中看到的代码：
MOV  BYTE PTR [label+1], 0x90  ; 运行时修改 label+1 处为 NOP
JMP  label
label:
DB   0xEB, 0x??                ; 文件里是 JMP 指令
                               ; 运行时被改成 NOP NOP
```

**绕过方法**：
```
1. 等自修改完成后再看代码（内存 dump）
2. 用硬件断点（不修改代码，靠 CPU 寄存器触发）
```

### 3.4 反调试检测层次

```ascii
                     用户层（API）
              ┌──────────────────────┐
              │ IsDebuggerPresent    │ ← 最容易被 Hook
              │ NtGlobalFlag         │ ← 比较容易被 Hook
              │ CheckRemoteDebugger  │ ← 也能被 Hook
              └──────────────────────┘
                         │
                     内核层（系统调用）
              ┌──────────────────────┐
              │ NtQueryInfoProcess   │ ← 内核层API，难Hook
              │ ZwQueryInfoProcess   │ ← 同上
              └──────────────────────┘
                         │
                    硬件层（无API调用）
              ┌──────────────────────┐
              │ RDTSC 时间差检测     │ ← 不需要API，最难Hook
              │ 异常处理检测          │ ← 不需要API
              │ 硬件断点检测          │ ← 不需要API
              └──────────────────────┘
```

**越靠下越难绕过**。

### 3.5 反反调试工具

| 工具 | 层级 | 功能 |
|:-----|:-----|:------|
| **ScyllaHide** | 用户层 | x64dbg 插件，Hook 常见反调试 API |
| **TitanHide** | 内核层 | 驱动级别的反反调试 |
| **SharpOD** | 用户层 | x64dbg 插件 |
| **手动 Patch** | 代码层 | 手工改汇编指令，最灵活 |

### 3.6 关于反调试的真相

> **反调试不是让软件无法被逆向，而是让逆向更慢、更费钱。**

程序员加了10层反调试 → 逆向者用ScyllaHide绕过5层，手动patch3层，研究2层 → 最终还是能破解。

商业软件的平衡：
- **破解成本 > 软件售价** → 没人愿意破解，赢了
- **破解成本 < 软件售价** → 被破解，亏了
