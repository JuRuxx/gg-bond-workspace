深度解析 Agent Memory 的架构设计，核心问题不是"Agent 怎么记住信息"，而是"过去的信息凭什么继续影响未来"。

文章首先澄清了几个容易混淆的概念：上下文窗口是当前工作集，Session 管当前会话连续性，Profile 是 Memory 的低维消费视图，Policy 是外部规则（Memory 不能改写）。Memory 的真正定义是：跨会话持续存在、可被更新和审计、并且会影响未来决策的结构化历史。

Memory 不能只围着用户偏好转。至少还有三类记忆同等重要：任务记忆（需求确认、方案否决、进度）、环境记忆（仓库结构、团队规则、CI 特点）、自我记忆（上次试过什么、哪些推断被证明是错的）。

写入 Memory 被理解为"预算分配"——不只是存储空间，还包括未来的检索成本、上下文成本、注意力成本。文章给出了写入的基本规矩：用户明确说的按 assertion 存，工具观察到的按 event 存，Agent 自己归纳的只能按 belief 存，未验证的必须保留"未确认"状态。

读取 Memory 不能只从 query 出发做相似度检索。应该先从任务上下文出发，弄清当前任务受什么约束，再去找对应的记忆。因为该影响当前决策的历史，未必和当前问题语义相似。

管理环节最容易被低估：冲突（用户去年说不喜欢 ORM 今年要求用 Prisma）、衰减（偏好有半衰期）、遗忘（系统必须知道旧信息什么时候不该再用）、安全（提示注入可以污染 Memory 并跨会话持续生效）。

对于 Coding Agent，文章建议分四层落地：当前工作集（上下文窗口）、工作区文件（AGENTS.md、GOAL.md 等）、memory store（跨 session 经验）、事件日志。最核心的原则是"人能读"——Memory 一旦出错，人要能查得动。
