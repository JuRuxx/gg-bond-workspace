# Agent Skills 的五种设计模式

## 摘要

本文梳理了 Agent Skills 从 Anthropic 解决 Skill 格式和加载机制，到 Google 提出 5 种设计模式的演进路径。核心观点是：Skill 不再只是"复用提示词"，而是可触发、可维护、可审查的过程资产，正在进入"该有自己设计模式语言"的阶段。Google 的 5 种 Skill 设计模式（Tool Wrapper、Generator、Reviewer、Inversion、Pipeline）分别解决了 Agent 在知识加载、输出稳定性、审查标准化、需求收集和复杂流程执行中的典型工程问题。文章进一步提出 Skills-Driven Development 理念、渐进式披露策略、Skill 治理框架，以及 Skill 作为"团队经验进入 Agent 运行时的接口"的定位。

## 详细解读

### 一、Agent 工程的演进脉络

文章将 Agent 工程划分为三个阶段的关注点转移：

1. **早期** — 如何写好一条 prompt
2. **中期** — 上下文如何组织、工具如何暴露、Subagent 如何隔离、MCP 如何接入外部系统
3. **现在（Skill 阶段）** — 团队经验、流程、清单、模板、排障方法开始被做成**模型可发现、可加载、可执行的工作单元**

Skill 在这个体系中的定位：Skill 是 Harness 运行时可以按需加载的过程模块。它与 CLAUDE.md（常驻上下文）、MCP（外部系统连接）、Subagents（上下文隔离）、Hooks（自动化强制约束）不是互相替代，而是互补组合。

> Kaxil Naik（Apache Airflow PMC member）的实践："Skill is the code" — 过去写在手里、脑子里、团队习惯里的工作方式，正在被写成 Agent 可执行的接口。

### 二、Skills-Driven Development

Zak El Fassi 提出的 SkDD 理念：每次构建循环里加一个判断——**"这件事，要不要变成一个 Skill？"** 如果是，写一个 SKILL.md，下次 Agent 自己就能找到、加载、执行。几个月后变成一份过程资产清单。

### 三、Google 的 5 种 Skill 设计模式

| 模式 | 解决的问题 | 核心思路 |
|------|-----------|----------|
| **Tool Wrapper** | Agent 不懂某个库/框架/规范 | 按需加载的领域手册 |
| **Generator** | 每次输出结构不稳定 | 带模板的填空流程 |
| **Reviewer** | 审查标准混乱 | 可替换的评分清单 |
| **Inversion** | Agent 没问清需求就开始生成 | 需求访谈器 |
| **Pipeline** | 复杂任务跳步骤 | 带检查点的工作流 |

### 四、构建 Skill 的 6 个关键问题

1. **何时触发？** — description 是路由契约不是简介，需要基于真实使用日志设定边界
2. **属于哪种模式？** — 先确定是知识注入、生成模板、审查结果、收集需求还是跑流程，再动笔
3. **哪些内容该拆出去？** — 稳定但很长的规范放 `references/`，输出模板放 `assets/`，确定性动放 `scripts/`；主文件只保留路由、流程、边界和加载规则。体现**渐进式披露**的价值。
4. **哪些步骤需要停下来？** — 生产级 Skill 需要检查点（需求没问完不生成方案、测试没跑过不宣称修复完成等）
5. **失败以后怎么走？** — 只写成功路径是最常见的问题。需要说明失败识别、证据收集、自动重试、人工介入、不可绕过的动作
6. **如何版本化和审查？** — 每个 Skill 有 owner、走 review、高风险 Skill 有测试样例、定期清理废弃规则、第三方 Skill 默认不信任

### 五、Skill 治理框架

文章强调了 Skill 治理的必要性：**坏提示词影响一次对话，坏 Skill 可能影响一串同类任务。**

| 来源 | 信任策略 |
|------|---------|
| 官方内置 Skill | 较高信任，仍要看版本 |
| 团队自写 Skill | 走 review、测试和变更记录 |
| Agent 自动生成 Skill | 默认草稿，需人工确认 |
| 社区第三方 Skill | 默认不信任，先做安全审查 |
| 带脚本的 Skill | 按可执行代码对待 |

作者最警惕的是：**错误经验被固化**。一次误判只在当前会话影响有限，如果被写进 Skill，以后每次类似任务都会更快走向同一个错误。

### 六、工程边界

- Skill 适合让 Agent 理解和应用流程
- 安全底线、危险命令阻断、权限控制、审计记录，更适合下沉到 Hook 等确定性更强的层
- Claude Code 文档的提醒："如果某条规则每次都要成立，更适合用 Hook 这类确定性机制强制"

### 七、核心观点总结

- **Skill 的本质**：是团队经验进入 Agent 运行时的一种接口，不是更长的提示词
- **设计模式的意义**：把这件事从"实验性玩法"拉进了"该有自己设计模式语言"的阶段
- **渐进式披露**：让 Agent 先知道有这项能力，再在真正需要时加载细节；上下文不是免费空间
- **description 是路由契约**：不是简介，是触发边界的定义
- **Agent 时代的架构师能力**：把作流拆成边界清楚、可组合、可观测、可回滚的系统部件
