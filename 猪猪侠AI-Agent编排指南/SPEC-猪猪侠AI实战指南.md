## 2. 写作规范

### 2.1 每章结构（强制）

```
## [知识点] — 用300-500字讲清楚核心概念

【我们的做法】— 800-1500字，真实案例，具体细节

【踩坑记录】— 200-400字，翻车故事，带教训

【猪猪侠补一刀】— 50-100字，金句/总结
```

### 2.2 风格要求

- **用大白话**。不要说"LLM的预训练目标是最小化交叉熵"，说"AI就是根据你上一句话，猜下一句哪个词最该出现"。
- **用真实数据**。比如"小呆呆写了1253行规格文档"、"超人强4分钟改了384篇笔记"、"迷糊老师从42篇抽查中发现12处问题"——这些比"提升500%效率"有说服力。
- **带情绪**。猪猪侠会吐槽、会自嘲、会得意。不要写成教科书。
- **引用来源精确**。说"吴恩达在2025年Agent Skills课程中讲到……"，不说"据吴恩达说……"。
- **内部引用用节标题**。不得出现"第X章"，一律用节标题引用（如"别紧张，参考【知识点】部分的说明即可"）。

### 2.3 禁止行为

- ❌ 禁止出现"第X章"引用
- ❌ 禁止模糊来源（"基于xxx经验"）
- ❌ 禁止纯理论——每章必须有我们的真实案例
- ❌ 禁止用表格（我们面向小白，表格不友好）
- ❌ 禁止用代码块（最多复制粘贴级别的命令）

**内容要点**：

   - **分工**：明确每个子Agent做什么

### 第9章：派活四步法（续）

**写作目标**：让读者掌握核心流程——需求→规格→内容→验收，理解这是"把人管住"的方法，不是"把AI变聪明"的方法。

**内容要点**：
1. **四步法的本质**：不是AI技巧，是信息压缩框架
   - 需求压缩目标空间（"帮我做个东西"→"做个能叠T恤的桌面机械臂"）
   - 规格压缩实现空间（用ROS2、YOLO、成本<5000）
   - 内容把概率云坍缩成现实产物
   - 验收强制现实校准
2. **每一步对抗什么混乱**
   - 需求→方向混乱（到底要什么？）
   - 规格→实现混乱（什么算对？）
   - 内容→执行混乱（真正产出了什么？）
   - 验收→幻觉混乱（现实里真的有效吗？）
3. **不要跳步**：省略规格=赌博，省略验收=自我感动

### 第11章：怎么写好Prompt？

**写作目标**：从"帮我写个XXX"升级到"按照规格写XXX"，让AI输出可预测、可控制。

**内容要点**：
1. **Prompt的进化史**
   - 第一阶段："帮我写篇文章"（AI自由发挥，质量看运气）
   - 第二阶段："帮我写篇文章，风格幽默，1000字左右"（加约束）
   - 第三阶段：写SPEC，让AI按规格生成（我们的做法）
2. **Prompt的底层逻辑**：AI是约束条件下的生成器，你给的约束越多，输出越可控
3. **从"问问题"到"给规格"的转变**
   - 问问题：依赖AI的知识库（你永远不知道它知道什么）
   - 给规格：你定义输入、输出、标准、约束（结果是可预测的）
4. **Anthropic提示词最佳实践中的关键原则**
   - 清楚（Be Clear）
   - 给例子（Give Examples）
   - 说不要什么（Negative Constraints）
   - 分步骤（Break Down）
5. **我们的Prompt演进史**：从"写篇文章"到"按SPEC写"的转变

**参考素材**：Anthropic官方演示_Claude提示词最佳实践_副本.md

### 第13章：验收的艺术

**写作目标**：让读者理解——没有验收的系统一定会漂移，验收不是找茬，是防漂移。

**内容要点**：
1. **为什么必须验收**
   - AI会偷懒（这是本性，不是恶意）
   - AI会幻觉（你给了错误的前提，它会沿着错误跑）
   - AI会漂移（长对话后，它会忘了一开始的约束）
2. **验收不是"查错"，是"校准"**
   - 没有验收的交付=自我感动
   - 验收的产出是"修正项"，不是"批评"
3. **怎么验收**
   - 对照规格逐条检查
   - 抽查（不用全部看，但要选代表性样本）
   - 让不同的人验收（执行者不验收自己的产出）
4. **验收发现了什么**（我们的翻车案例）：
   - tags只给1-2个而不是3-6个
   - aliases直接复制文件名而不是写有意义的别名
   - 脉冲目录质量不达标
   - "世界模型 1.md"被漏掉
5. **验收后的返工机制**
   - 返工不是惩罚，是迭代
   - 每次返工都是系统的质量提升

**参考素材**：Agent落地问题3_产品组织与信任_副本.md

**字数**：3000-4000字

### 第14章：搭建信息雷达

**写作目标**：教读者用AI建立自己的信息采集系统——从B站、微信、公众号抓高质量信息，而不是被算法投喂。

**内容要点**：
1. **信息爆炸时代的困境**：不是信息太少，是高质量信息被噪音淹没了
2. **AI剪报方法论**
   - 订阅源（B站关注的UP主、微信公众号、RSS）
   - AI摘要（给AI指令：提取核心观点、保留原文细节、标注来源）

### 第17章：从复利到幂塔

**写作目标**：把全书升华——你的AI团队不是一次性的，是越用越强的。

**内容要点**：
1. **复利的本质不是结果累积，是减少不确定性**
   - 有SOP、checklist、benchmark、failure case
   - 第二次开始时，系统的先验概率已经改变了
2. **幂塔的四层跃迁**
   - 线性：人→结果（一次性劳动）
   - 复利：人→流程→更好的结果（经验沉淀为资产）
   - 闭环：人→流程→更强的自己→更强的流程（流程反过来增强人）
   - 幂塔：系统→增强系统本身（Agent自动复用历史、发现最佳规格、优化workflow）
3. **幂塔的前提条件**
   - 标准化（可复制）
   - 验收（可校准）
   - 模块化（可组合）
   - 记录（可积累）
   - 反馈（可优化）
   - 自动化（可扩张）
   - 现实校准（防漂移）
4. **最大的危险：错误也会幂塔化**
   - 验收错了 → 错误越积累越大
   - 指标错了 → 系统高速偏离现实
   - 现实校准能力 ≥ 增长速度，否则增长越快崩得越快
5. **最后的总结**：管理最核心的是人性约束工程，AI团队也是

## 9. 全书写完后的验收标准

每次写完一批章节（比如第一卷5章写完后），迷糊老师进行统一验收。

### 9.1 内容完整性（每章必查）

- [ ] 每章有【知识点】+【我们的做法】+【踩坑记录】+【猪猪侠补一刀】四个模块完整
- [ ] 每章至少有一个**真实案例**（我们的实际经历，带具体数字/细节）
- [ ] 没有纯理论段落（每一个理论点后面都有我们怎么做的例子）
- [ ] 来源精确：引用外部资料注明来源（作者/课程/文章标题）
- [ ] 字数符合规格（误差不超过±500字）

### 9.2 风格检查

- [ ] 没有"第X章"引用（统一用节标题引用）
- [ ] 没有代码块（最多一行命令级别的复制粘贴）
- [ ] 没有表格（用列表代替）
- [ ] 语言口语化，有猪猪侠的味道——读起来像朋友在聊天，不是老师在讲课
- [ ] 自嘲但不失自信（猪猪侠的典型腔调）
- [ ] 每章至少有一个让人"哈哈"或"卧槽"的瞬间

### 9.3 可执行性检查（面向AI小白）

- [ ] 第2-5章的命令和步骤全部经过验证（命令是否正确、是否遗漏前置步骤）
- [ ] 每一步都有对应的预期结果（"当你看到XXX说明成功了"）
- [ ] 避坑提示到位（常见错误有预警）
- [ ] 没有假设读者"应该知道"的知识（假设读者是从零开始）
- [ ] 没有使用未解释的专业术语（如果必须用，第一次出现时加括号解释）

### 9.4 跨章一致性检查

- [ ] 各章之间没有矛盾的说法（比如第2章说"Mac mini够了"，第5章又说"需要GPU"）
- [ ] 术语使用一致（同一概念在各章中叫法统一）
- [ ] 各章的语气和风格一致（虽然是不同子Agent写的，但最终读起来像一个人写的）
- [ ] 整体结构连贯（章节之间有过渡，不是各写各的）

### 9.5 质量门槛

- **合格**：9.1-9.4 所有项目通过率 ≥ 80%
- **优秀**：9.1-9.4 所有项目通过率 ≥ 95%
- **返工**：低于80%，回到作者手中修改后重新验收
- **批次验收**：每写完一个批次（如第一卷5章）就验收一次，不要全部写完才验收
