"真实的 Agent 落地"三部曲第三篇，聚焦产品定义、组织能力和用户信任等非技术问题。

一个反直觉的观察：Agent 项目失败的原因中，纯技术问题约占 30%，剩下 70% 是产品定义错误、组织能力不匹配、商业模式不成立、用户信任建立失败等"非技术问题"。

产品定义的陷阱："Agent"不是产品，"帮用户完成 XX 任务"才是。最常见的死法是做了一个"什么都能做但什么都做不好"的东西。用户想要的不是"完全自动化"，而是"在我信任的范围内自动化"——但这个信任范围因人而异、因场景而异，而且用户自己也说不清楚。

信任构建遵循残酷规律：建立信任需要 100 次成功，摧毁信任只需要 1 次失败。更糟糕的是，信任归零后几乎不可能恢复——用户会一直带着"它可能又搞错了"的心态使用，效率优势荡然无存。同时，过度信任同样危险——用户停止检查 Agent 输出后，错误不会被及时发现和纠正。

透明度悖论：过于透明会降低信任（用户看到 Agent 犹豫纠结的痕迹会觉得不靠谱），不够透明也会降低信任（结果不符合预期时用户质疑"你凭什么这么做"）。选择性透明是正确方向，但极难把握度。

Demo 驱动开发是最甜蜜的毒药。Demo 效果太好设定了所有人的预期，但 Demo 到生产环境的距离远超想象——Demo 覆盖 3 个场景，生产环境要覆盖 300 个。Demo 完成了 5%，不是 80%。
