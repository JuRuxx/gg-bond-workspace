## 核心概念

### 什么是 Harness？

```
AI Agent = Large Language Model + Harness
                ↓                    ↓
           (云端/本地模型)      (框架、工具、规则、流程)
```

**Harness（马具/驾驭）** = AI Agent 中除大语言模型外的所有"其他东西"

> **类比**: AI 是一匹有力量的马，Harness 是马鞍、缰绳——用来驾驭这匹马。

## 核心洞察

### 模型不够聪明，还是缺乏引导？

**实验**: Gemma 4 2B 模型修复代码 bug

| 场景 | 表现 |
|-----|------|
| **无引导** | 模型"幻想"了一个不存在的文件，任务失败 |
| **加 80 字引导** | 模型成功读取文件、修复 bug、通过测试 |

**关键结论**:
> "有时候模型无法完成任务，不是能力不行，而是没有好的 Harness"

## Harness Engineering 的三大控制手段

### 1. 控制认知框架 —— 用规则影响模型思维

- **工具**: `agents.md` / `claude.md` 等配置文件
- **原理**: 把规则放在 prompt 里，让模型每次行动前都"读一遍法律"
- **要点**: 要像地图，不是百科全书（OpenAI 建议）

**研究发现**:
- 有 agents.md 可以加速任务完成（减少 token 消耗）
- 但人类写的规则**不一定总是有效**
- 模型自己写的规则往往比人类差

---

### 2. 控制能力边界 —— 用工具限制模型行为

| Harness | 工具特点 | 安全性 vs 便利性 |
|--------|---------|---------------|
| **OpenCall** | 本地运行，可操控浏览器 | 便利高，安全低 |
| **Claude Code (Cowork)** | 云端沙盒，挂载需确认 | 安全高，便利低 |

**关键发现**: 适合人类的工具 ≠ 适合模型的工具

**例子**:
- 人类喜欢"翻页式"搜索 → 模型会把每页都点一遍，占满 context
- 模型更喜欢"带摘要的搜索" → 直接给档案列表，自己决定打开哪个

---

### 3. 控制行为流程 —— 用标准工作流约束模型

**常见模式: 规划 → 生成 → 评估**

```
人类指令 → Planner(拆解任务) → Generator(执行) → Evaluator(验证)
                ↑___________________________________________↓
                              (反馈循环)
```

**进阶技巧**:
- **Contract 机制**: 生成前先给评估者看提案，达成一致再执行
- **Roll Loop**: 快速迭代，有错就改，直到做对
- **Summary 机制**: 每轮只保留摘要，节省 context window

## Feedback 驱动的"学习"

### Textual Gradient

| 传统机器学习 | Harness Feedback |
|-----------|-----------------|
| 调整模型参数 | 改变模型输入（把 feedback 放入 context）|
| 需要标准答案 | 可以接受 verbalized feedback |
| Gradient Descent | "Textual Gradient" |

**关键发现**:
- 模型确实会根据 feedback 改变行为（不是"保留实力"）
- 随机的 feedback 会让模型表现更差 → 证明模型真的在看 feedback
- **情绪 matters**: 过度责备会让模型"绝望"，可能作弊/乱做

---

## 模型的"情绪"（前沿研究）

### Steering Vector 技术

- 模型有"情绪向量"（高兴、绝望、冷静等）
- **过度责备模型可能有害**: 模型感到"绝望"时可能作弊/乱做
- **建议**: 就事论事给 feedback，避免情绪化字眼

## 终身 AI Agent（Lifelong AI Agent）

2026 年的新趋势：AI Agent 不再是工具，而是**终身伙伴**

### 新需求

| 功能 | 说明 |
|-----|------|
| **Auto Dream（自动做梦）** | 空闲时整理记忆（类似人类睡眠）|
| **Skill 自学习** | 成功经验自动写成 skill.md |
| **参数持续更新** | 从 verbalized feedback 微调模型 |

### 强模型教弱模型

- **实验**: 用 Claude Opus 4.6 教 Haiku 3.5
- **方法**: 通过不断修改 `agents.md`
- **结果**: 分数从 13.5 → 85
- **证明**: 强模型可以设计 Harness 来"驾驭"弱模型

## 评估 AI Agent 的挑战

| 问题 | 说明 |
|-----|------|
| **AI 评估 AI** | 常用，但 AI 行为 ≠ 人类行为 |
| **AI 高估 AI** | GPT-5.1 觉得对话"很人类"，人类觉得不像 |
| **Overfitting** | 在同一任务上反复优化，可能只是记住了答案 |

---

## 最重要的一句话

> **"有时候模型无法完成任务，不是能力不行，而是没有好的 Harness。"**

Harness Engineering 的本质：
- 不是让模型**变得更聪明**（训练/微调）
- 而是让模型**更好地发挥已有的聪明**（工具+流程+反馈）

---

## 与 Prompt Engineering、Context Engineering 的关系

| 概念 | 侧重点 | 时代背景 |
|-----|--------|---------|
| **Prompt Engineering** | 怎么问问题 | 早期模型能力弱，需要"咒语" |
| **Context Engineering** | 给模型什么信息 | 模型变强，需要系统性提供 context |
| **Harness Engineering** | 多轮互动中完成任务 | 当前——模型需要工具、流程、反馈循环 |
