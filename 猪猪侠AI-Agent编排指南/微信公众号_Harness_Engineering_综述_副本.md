## 三、Harness的核心组件

### 3.1 六大核心组件（黄佳）

| 组件 | 功能 | 类比 |
|-----|------|------|
| **Agentic Loop** | 接受输入→循环执行→返回结果 | 心脏 |
| **Tool System** | 工具调用能力 | 手脚 |
| **Memory & Context Management** | 上下文管理 | 记忆 |
| **Guardrails** | 权限控制（Allow/Deny/Ask） | 缰绳 |
| **Hooks** | 安全检查 | 守卫 |
| **Session** | 会话连续性 | 状态 |

### 3.2 两类控制机制（Birgitta Böckeler框架）

| 类型 | 名称 | 作用时机 | 作用 | 示例 |
|-----|------|---------|------|------|
| **前馈控制** | Guides | 行动之前 | 防患于未然 | CLAUDE.md、代码规范、架构决策 |
| **反馈控制** | Sensors | 行动之后 | 亡羊补牢 | 自动化测试、代码lint、CI流水线 |

## 四、Harness解决的五大难题

1. **无限循环问题** —— 通过Agentic Loop控制
2. **上下文爆炸问题** —— 通过Context Management解决
3. **权限失控问题** —— 通过Guardrails控制
4. **质量不可控问题** —— 通过Sensors检测
5. **成本不透明问题** —— 通过系统监控解决

## 五、12节课渐进式构建（龙井Agent）

### 5.1 课程架构

| 阶段 | 课节 | 核心机制 | 解决的问题 |
|-----|------|---------|-----------|
| **让它跑起来** | S01 | 一个循环 | 智能体能运转 |
| | S02 | 工具箱 + 围栏 | 能干活，也有边界 |
| **让它干得好** | S03 | Todo清单 | 不跑偏、注意力不漂移 |
| | S04 | 子智能体 | 分工、上下文隔离 |
| | S05 | 技能文件夹 | 知识体系、按需加载 |
| | S06 | 三层压缩 | 无限会话、不爆上下文 |
| **让它记得住** | S07 | 任务图 | 关机了任务还在 |
| | S08 | 后台线程 | 不傻等、并行处理 |
| **让它带团队** | S09 | 队友 + 收件箱 | 分工协作 |
| | S10 | 通信规矩 | 关机要握手、大事要审批 |
| | S11 | 自动认领 | 不用老板盯着、自己找活 |
| | S12 | 独立目录(git worktree) | 各写各的、互不干扰 |

### 5.2 关键设计详解

#### S05 技能加载机制（Lazy Loading）
```
启动时：只加载技能名称和描述（几十token）
执行时：调用load_skill才加载完整内容（几千token）

核心思想：启动时只给菜单，点菜时才上完整食谱
```

#### S06 三层上下文压缩
1. **自动替换**：3轮以前的内容替换成标记 `[Previous: used read_file]`
2. **阈值触发**：超token时存盘+摘要替换
3. **主动压缩**：智能体自己调用压缩工具

#### S07 任务系统（图结构）
- 不是线性队列，而是**依赖图**
- 任务状态：pending → in_progress → completed
- 自动解锁：前置任务完成→后续任务自动开始

#### S12 Worktree隔离
```
任务ID ←→ Git Worktree ←→ 独立分支
  ↓         ↓           ↓
任务系统   工作空间      干净回滚
```

## 七、工程师能力转型

### 从码农到系统工程师

| 维度 | 码农 | 工程师 |
|-----|------|--------|
| **核心工作** | 写代码 | 设计并驾驭复杂系统 |
| **能力要求** | 编码技能 | 抽象思维、系统理解 |
| **AI时代价值** | 可被替代 | 不可替代 |
| **关键能力** | 语法掌握 | 驾驭不确定性 |

### 核心能力

1. **理解系统的复杂性**
2. **抽象和结构化思维**
3. **驾驭不确定性**
4. **深度交互能力**
5. **业务理解（护城河）**
