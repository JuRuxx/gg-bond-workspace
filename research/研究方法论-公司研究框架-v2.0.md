## 一、研究流程（四步法）

### Step 1: 需求（PRD）
**目标**：明确研究目的和验收标准

**模板**：
```markdown
# [公司名称] 研究 - PRD

## 研究目标
- 主要目标：[例如：拆解商业模式，为自研产品提供参考]
- 次要目标：[例如：评估投资价值、学习对标]

## 核心问题
1. [问题1：例如 公司怎么赚钱的？]
2. [问题2：例如 商业模式的优势和坑在哪？]
3. [问题3：例如 我们能借鉴什么？避开什么？]

## 验收标准
- [ ] 完整的商业模式拆解
- [ ] 财务数据分析（趋势+问题）
- [ ] 可复用的策略清单
- [ ] 风险预警清单

## 研究范围
- [ ] 公司概览
- [ ] 商业模式
- [ ] 财务数据
- [ ] 团队背景
- [ ] 技术架构
- [ ] 客户案例
- [ ] 竞争分析

## 数据保留要求
- [ ] 搜索执行记录（search-records/）
- [ ] 原始数据归档（源数据/）
- [ ] 数据来源表（data-sources/）
- [ ] 中间分析文件

## 不包含
- [明确排除的内容]
```

---

### Step 2: 规格（SPEC）
**目标**：定义报告结构和质量标准

**模板**：
```markdown
# [公司名称] 研究 - SPEC

## 报告结构

### 第一章：公司概览
- 1.1 基本信息（名称、代码、上市日期、定位）
- 1.2 IPO表现（发行价、首日表现、市值、认购热度）
- 1.3 产品矩阵（产品名、定位、收入占比）

### 第二章：商业模式深度拆解
- 2.1 核心产品收费模式
- 2.2 收入结构分析
- 2.3 商业模式对比（与传统模式对比）

### 第三章：财务健康度诊断
- 3.1 营收趋势（3-5年）
- 3.2 利润趋势（核心问题）
- 3.3 毛利率趋势
- 3.4 其他关键指标（客户集中度、留存率等）

### 第四章：团队背景与融资历程
- 4.1 创始人团队（教育背景、职业经历、特点）
- 4.2 融资历程（轮次、金额、估值、投资方）
- 4.3 主要投资方（持股比例、背景）
- 4.4 IPO历程（冲刺次数、路径选择）

### 第五章：技术架构与核心能力
- 5.1 技术定位
- 5.2 产品技术架构
- 5.3 核心技术能力
- 5.4 关键技术创新
- 5.5 行业Know-How积累
- 5.6 技术演进路径
- 5.7 核心竞争优势

### 第六章：客户案例深度分析
- 6.1 客户痛点
- 6.2 解决方案（技术架构+功能模块）
- 6.3 量化效果（核心指标提升）
- 6.4 关键成功因素
- 6.5 行业认可

### 第七章：优劣势分析
- 7.1 优势（可借鉴点）
- 7.2 劣势（需避开的坑）

### 第八章：可借鉴策略
- 8.1 商业模式设计
- 8.2 产品策略
- 8.3 定价策略
- 8.4 技术策略

### 第九章：风险预警
- 9.1 商业模式风险
- 9.2 行业风险
- 9.3 财务风险
- 9.4 估值泡沫分析

### 第十章：总结与行动建议
- 10.1 核心启示
- 10.2 行动指南（做/不做清单）
- 10.3 关键指标监控

## 输出格式
- Markdown结构化报告
- 包含数据表格和对比分析
- 每个章节都有"对我们的启示"

## 质量标准
- 数据准确（来源于招股书、财报、权威媒体）
- 分析深入（不仅描述现象，还要解释原因）
- 可操作（提供具体的行动建议）
- 风险预警（明确指出潜在风险）
- 批判性思维（质疑估值、商业模式可持续性）
```

### Step 3: 内容（CONTENT）
**目标**：执行研究，生成报告

**搜索策略**（串行执行，避免并发失败）：

#### 第一轮搜索：基础信息
```
搜索1：[公司名称] 招股书 营收 利润 毛利率 财务数据
搜索2：[公司名称] 商业模式 收费模式 客户案例
搜索3：[公司名称] 创始人 团队背景 融资历程 投资方
```

#### 第二轮搜索：深度信息
```
搜索4：[公司名称] 核心技术 竞争优势 技术架构
搜索5：[公司名称] [核心产品名] 客户案例 效果
搜索6：[公司名称] 行业分析 竞争格局 市场地位
```

#### 第三轮搜索：验证和补充
```
搜索7：[公司名称] 风险 问题 质疑
搜索8：[公司名称] 最新动态 新闻 报道
```

**搜索关键词优化**：
- 使用中文关键词
- 关键词数量控制在5-8个
- 避免过长的查询语句
- 优先使用专业术语（如"招股书"而非"IPO文件"）

**数据验证**：
- 财务数据：优先使用招股书、财报数据
- 案例数据：优先使用官方发布或权威媒体报道
- 技术指标：要求有明确的量化数据

**批判性分析**：
- 质疑估值合理性（P/E、P/S等指标）
- 分析商业模式可持续性
- 识别"故事"与"现实"的差距
- 评估财务健康度（增收不增利？）

**写作质量保障**：写作相关的质量标准（三要素清单、质量标准卡、好句案例库、自我复盘）已独立成文档，参见：
> [`${RESEARCH_DIR}/写作方法论-质量保障标准.md`](../写作方法论-质量保障标准.md)

## 二、研究维度清单（按优先级）

### 必选维度（每次研究必做）
1. **商业模式**：怎么赚钱？收费模式？
2. **财务数据**：营收、利润、毛利率趋势
3. **优劣势分析**：可借鉴点 vs 需避开的坑
4. **风险预警**：商业模式风险、财务风险、估值泡沫

### 重要维度（根据研究目的选择）
5. **团队背景**：创始人、融资历程、投资方
6. **技术架构**：核心技术、竞争优势
7. **客户案例**：痛点、解决方案、量化效果
8. **竞争分析**：市场地位、竞争格局

### 可选维度（深度研究时补充）
9. **行业趋势**：行业发展方向、政策影响
10. **估值分析**：P/E、P/S、DCF等估值方法
11. **国际化**：海外业务、出海策略
12. **ESG**：环境、社会、治理

## 三、不同类型公司的研究重点

### SaaS公司
**重点关注**：
- ARR（年度经常性收入）
- NRR（净收入留存率）
- CAC（客户获取成本）
- LTV（客户终身价值）
- 毛利率（应该>60%）
- Rule of 40（收入增长率+利润率>40%）

**警惕信号**：
- 毛利率<60%（不是纯SaaS）
- NRR<100%（客户在流失）
- CAC回收期>18个月

---

### 平台型公司
**重点关注**：
- GMV（平台交易总额）
- Take Rate（平台抽成比例）
- 网络效应（双边/多边）
- 用户活跃度（DAU/MAU）
- 转换成本

**警惕信号**：
- 用户增长放缓
- Take Rate提升困难
- 竞争者涌入

---

### 服务型公司
**重点关注**：
- 人效（人均收入）
- 项目毛利率
- 客户复购率
- 标准化程度
- 可规模化能力

**警惕信号**：
- 毛利率<30%
- 人效低
- 难以标准化

---

### AI公司
**重点关注**：
- 技术壁垒（专利、算法、数据）
- 落地案例（真实效果、ROI）
- 数据飞轮（是否越用越好）
- 算力成本
- 商业化路径

**警惕信号**：
- 只有技术没有落地
- 案例太少
- 增收不增利
- 过度依赖大客户

---

## 四、泡沫检测清单

### 估值泡沫信号
- [ ] P/E > 100倍
- [ ] P/S > 20倍
- [ ] 市值/营收 > 行业平均3倍以上
- [ ] IPO首日涨幅 > 100%
- [ ] 超额认购 > 1000倍

### 商业模式泡沫信号
- [ ] 增收不增利（营收增长但利润下滑）
- [ ] 毛利率持续下滑
- [ ] 客户集中度高（前5大客户>50%）
- [ ] 现金流为负
- [ ] 过度依赖单一业务

### 故事泡沫信号
- [ ] 概念炒作（AI、区块链、元宇宙等）
- [ ] 落地案例少但估值高
- [ ] 创始人过度营销
- [ ] 媒体报道过度正面
- [ ] 缺乏第三方验证

### 财务泡沫信号
- [ ] 应收账款增速>营收增速
- [ ] 存货周转率下降
- [ ] 关联交易占比高
- [ ] 审计意见非标
- [ ] 频繁更换审计机构

## 六、研究效率提升技巧

### 搜索技巧
1. **串行执行**：避免并发搜索被中断
2. **关键词精简**：5-8个关键词，避免过长
3. **多轮搜索**：第一轮基础信息，第二轮深度信息，第三轮验证补充
4. **优先权威来源**：招股书、财报、权威媒体
5. **交叉验证**：同一数据至少2个来源确认

### 分析技巧
1. **数据驱动**：每个结论都要有数据支撑
2. **对比分析**：与行业平均、竞争对手对比
3. **趋势分析**：看3-5年趋势，而非单点数据
4. **批判性思维**：质疑估值、商业模式可持续性
5. **第一性原理**：从本质出发，不被表面现象迷惑

### 写作技巧
1. **结构化**：使用Markdown标题、列表、表格
2. **可视化**：用表格对比数据，用列表总结要点
3. **精炼**：每个段落只讲一个要点
4. **可操作**：提供具体的行动建议
5. **批判性**：指出风险和问题，不只讲好话

## 七、常见陷阱与规避

### 陷阱1：技术导向，忽视商业本质
**表现**：过度关注技术细节，忽视商业模式和财务健康度
**规避**：先问"公司怎么赚钱"，再看"技术有多牛"

### 陷阱2：被故事迷惑，忽视数据
**表现**：被"AI"、"区块链"等概念吸引，忽视财务数据
**规避**：用泡沫检测清单验证，关注P/E、毛利率、现金流

### 陷阱3：只看优点，忽视风险
**表现**：只讲好话，不分析风险和问题
**规避**：必须包含"风险预警"章节，质疑估值和商业模式

### 陷阱4：案例太少，过度推断
**表现**：只有1-2个案例就推断"技术很牛"
**规避**：要求至少5个案例，且有效果数据支撑

### 陷阱5：忽视行业背景
**表现**：脱离行业背景分析公司
**规避**：先了解行业趋势、竞争格局，再分析公司

## 八、研究质量评分卡

| 维度 | 权重 | 评分标准 |
|:---|:---|:---|
| **数据准确性** | 20% | 数据来源权威，交叉验证 |
| **分析深度** | 20% | 不仅描述现象，还解释原因 |
| **可操作性** | 15% | 提供具体的行动建议 |
| **风险预警** | 15% | 明确指出潜在风险 |
| **批判性思维** | 15% | 质疑估值、商业模式可持续性 |
| **结构化程度** | 10% | Markdown格式清晰，表格对比 |
| **完整性** | 5% | 覆盖所有必选维度 |

**评分**：
- 90-100分：优秀，可直接用于决策
- 80-89分：良好，可作为参考
- 70-79分：合格，需要补充
- <70分：不合格，需要重做

## 九、数据保留规范

> 学术级研究要求每条结论可回溯、可验证。所有原始数据和中间产物必须完整保留。

### 9.1 数据保留目录结构

```
${RESEARCH_DIR}/
├── [公司名称]-PRD-*.md          # 需求文档（研究目标、核心问题）
├── [公司名称]-SPEC-*.md         # 规格文档（报告结构、质量标准）
├── [公司名称]-深度研究报告-*.md  # 主报告（最终产出）
├── [公司名称]-REVIEW-*.md       # 验收文档（质量检查、经验总结）
├── 源数据/                       # 原始外部资料
│   ├── [公司名称]-招股书-摘要.md
│   ├── [公司名称]-财报-2024.md
│   ├── [公司名称]-财报-2025.md
│   ├── [公司名称]-行业报告-引用.md
│   └── ...
├── search-records/               # 搜索执行记录
│   ├── [公司名称]-搜索记录.md    # 每次搜索的关键词、结果摘要URL
│   └── ...
└── data-sources/                 # 数据来源索引
    └── [公司名称]-数据来源表.md   # 报告中每条数据的来源、获取时间、验证状态
```

### 9.2 搜索记录规范

每次搜索必须保留以下信息：

| 字段 | 说明 |
|:---|:---|
| **搜索轮次** | 第几轮搜索（基础/深度/验证） |
| **搜索关键词** | 实际使用的查询语句 |
| **搜索时间** | 搜索执行的日期时间 |
| **搜索结果摘要** | 返回的关键结果 |
| **引用URL** | 使用的信息源链接 |
| **数据使用** | 是否写入报告、用于哪个章节 |
| **验证状态** | 该数据是否经过交叉验证 |

### 9.3 数据来源表规范

每条写入报告的数据，必须记录：

| 字段 | 说明 |
|:---|:---|
| **数据描述** | 具体数据项（如：2025年营收5.77亿元） |
| **原始来源** | 数据最初出处（招股书/财报/行业报告/媒体报道） |
| **引用URL/文件** | 可回溯的具体链接或本地文件路径 |
| **获取时间** | 数据获取的日期 |
| **报告章节** | 该数据出现在报告的哪个章节 |
| **验证方式** | 是否经过交叉验证，验证结果 |
| **备注** | 数据备注（如口径说明、误差范围） |

### 9.4 数据保留原则

1. **不删减**：原始搜索记录和中间分析文件不因"报告完成"而删除
2. **可回溯**：报告中的每个结论和数据点，必须能追溯到原始来源
3. **统一格式**：搜索记录和来源表使用统一模板，便于后续检索
4. **版本保留**：搜索记录和来源表随报告版本更新，不覆盖旧版
