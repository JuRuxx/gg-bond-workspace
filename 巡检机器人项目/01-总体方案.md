## 0. 导航方案选择

> 2026-05-27 补充：老板提议考虑**磁轨导航**替代纯视觉方案

巡检机器人导航方式对比：

### 方式A：视觉导航（原方案）
- **原理**：摄像头拍摄地面/环境特征，通过OpenCV/SLAM定位
- **成本**：高（摄像头+算力≈¥500-1,000，树莓派/Jetson）
- **精度**：±5mm
- **灵活性**：高——路径可随时改，只需要重新录轨迹
- **弱点**：怕光线变化（强光/阴影/灰尘影响识别）
- **适用**：路径可能变化、环境光照稳定的场景

### 方式B：磁条导航 🧲（老板说的磁轨）
- **原理**：地面贴磁条，机器人底部磁传感器沿着走
- **成本**：极低（磁条几元/米，磁传感器¥20-50）
- **精度**：±20mm
- **灵活性**：低——改路径要重新贴磁条
- **弱点**：磁条易被叉车压坏、油污覆盖
- **适用**：固定路线、环境不太恶劣的产线

### 方式C：磁钉导航 📍
- **原理**：地面埋磁钉，机器人经过时磁传感器感应定位
- **成本**：中（磁钉+施工¥200-500）
- **精度**：±10mm
- **灵活性**：低——改路径要重新埋钉
- **弱点**：施工麻烦
- **适用**：需要耐用、高精度固定路线的场景

### 推荐建议
| 因素 | 选视觉 | 选磁轨 |
|:----|:------|:------|
| 预算充足 | ✅ | - |
| 路径可能会改 | ✅ | ❌ |
| 环境有灰尘/油污 | ❌ | ✅ |
| 需要低维护 | ❌（摄像要擦） | ✅ |
| 要跑多种路线 | ✅ | ❌ |

**混合建议**：如果路线固定，可以**磁条导航为主 + 单摄像头辅助定点识别**（只在重要工位拍照分析），这样成本低又兼顾了视觉的检测能力。

## 3. 系统工作流程

### 完整循环

```
 ┌─────────────── 开机自检 ───────────────┐
 │ 主控上电 → 传感器自检 → WiFi连接       │
 │ → 定位校准（航向/位置初始化）           │
 │ → 等待远程启动指令 / 定时任务触发       │
 └──────────────────┬────────────────────┘
                    ▼
 ┌─────────── 沿产线巡检 ─────────────────┐
 │ ① 沿预设路径前进（基于编码器里程计）   │
 │ ② 超声波实时避障 → 遇到障碍降速/绕行  │
 │ ③ 到达定点站（工位/设备） → 停车       │
 └──────────────────┬────────────────────┘
                    ▼
 ┌─────────── 采集与分析 ─────────────────┐
 │ ④ 摄像头拍照 + OpenCV分析              │
 │    - 仪表盘读数识别                     │
 │    - 设备状态指示灯判断（红/绿/黄）     │
 │    - 漏液/异物检测                      │
 │ ⑤ MLX90614 红外测温 → 关键节点温度     │
 │    - 超过阈值 → 立即报警                │
 │    - 接近阈值 → 记录+标记复查           │
 │ ⑥ 超声波测距 → 确认设备间距正常        │
 └──────────────────┬────────────────────┘
                    ▼
 ┌─────────── 结果处理 ───────────────────┐
 │ 无异常 → 写入日志 → 继续下一站         │
 │ 有异常 → 蜂鸣器+LED闪烁                │
 │        → 拍照上传至服务器               │
 │        → MQTT/HTTP推送通知（钉钉/微信）│
 │        → 等待远程指令（原地等/继续/返航）│
 └──────────────────┬────────────────────┘
                    ▼
 ┌─────────── 循环与回充 ─────────────────┐
 │ 当前电量 > 阈值（如20%）→ 继续下一站   │
 │ 当前电量 ≤ 阈值                         │
 │ → 完成当前站 → 沿最短路经回桩           │
 │ → 红外对位 → 触点充电                   │
 │ → 充满后继续巡检 / 进入待机             │
 └─────────────────────────────────────────┘
```

### 关键时序约束

| 动作 | 目标耗时 | 说明 |
|:----|:-------:|:----|
| 单站巡检周期 | ≤ 30 秒 | 含停车→拍摄→分析→判断→记录 |
| 避障响应 | ≤ 200 ms | 超声波测距→减速/转向 |
| 异常报警 | ≤ 1 秒 | 检测到异常→蜂鸣+推送 |
| 回桩对位 | ≤ 30 秒 | 从2m外开始→对准→接触充电 |

## 4. 软件架构

### 4.1 主控操作系统推荐

**推荐：Raspberry Pi OS（Debian） + ROS 2（Humble）**

理由：
- ROS 2 提供成熟的**导航栈（Nav2）**、**SLAM建图**、**里程计融合**
- 节点化架构让各个模块解耦，调试方便
- 可平滑升级到Jetson Nano（ARM架构兼容）
- 社区资料丰富，问题秒搜秒解

若觉得ROS太重、只想快速跑原型，也可选择**裸机Python主循环**：
- 用 `threading` 或 `asyncio` 实现多路采集
- 省掉ROS学习成本，但后续功能扩展和维护难度会上升

### 4.2 关键软件模块

```
巡检机器人软件系统
├── master_node          ← 主调度节点（状态机：IDLE→PATROL→CHARGING→ALARM）
```

### 4.3 通信协议

| 层级 | 协议 | 用途 | 数据格式 |
|:----|:----|:-----|:--------|
| 设备间（I²C） | I²C | MLX90614测温读取 | 16位温度值 |
| 设备间（GPIO） | PWM | 电机速度控制、舵机 | 占空比信号 |
| 设备间（UART） | 串口 | 激光雷达/其他串口传感器 | 二进制/NMEA |
| 网络通信 | **MQTT** | 常规巡检数据上报（轻量） | JSON (`{station, temp, status}`) |
| 网络通信 | HTTP/WebSocket | 图片上传、远程控制指令 | multipart/form-data |
| 网络通信 | OTA | 固件升级 | HTTP下载 .tar.gz |

**MQTT Topic 设计示例：**

```
robot/{id}/status       ← 上报：运行状态、电量
robot/{id}/telemetry    ← 上报：温度、位置、传感器数据
robot/{id}/alarm        ← 上报：异常报警
robot/{id}/command      ← 订阅：远程控制指令（返回/暂停/继续）
robot/{id}/image        ← 上报：异常图片URL
```

## 6. 风险点与对策

### ⚠️ 风险 1：回桩对准精度

**问题**：机器人从不同位置、角度返回充电桩时，触点对接可能偏移 3-5cm，导致接触不良。

**对策**：
- 回桩路径分两段：粗定位（里程计）→ 精对位（红外/激光传感器引导最后 1 米）
- 充电桩触点设计成 V 型导槽 + 可浮动触点，允许 ±3cm 偏差
- 触点背面加磁铁，辅助最后吸附对准
- 实在不行加装一个机械导引臂（成本约¥30）

### ⚠️ 风险 2：视觉检测受光照影响

**问题**：工厂产线照明条件复杂——强光、阴影、反光表面都会影响OCR和颜色识别准确率。

**对策**：
- 摄像头旁加装白色LED补光灯（恒亮/触发补光），¥15 解决
- 采集阶段做光照归一化预处理（直方图均衡化、自适应阈值）
- 关键位置（仪表盘）用模板匹配 + 局部增强，而非全局识别
- 阴天/夜间也不怕——补光灯+红外滤光片摄像头可全天候工作
- 入库前在目标光照下采集3组以上训练数据

### ⚠️ 风险 3：红外测温精度受距离影响

**问题**：MLX90614 最佳测量距离 5-10cm，巡检机器人停在 20-30cm 外时读数偏差可达 2-5°C。

**对策**：
- 物理固定：将红外传感器安装在可调支架上，电机带动伸出至目标距离（加个微型舵机+连杆）
- 或者：在软件层做**距离补偿曲线**——通过实验标定不同距离下的温度偏差，建立修正函数
- 对低于报警阈值的数据宽松判断（比如设定报警阈值=50°C，实际在45°C时即标记复查）
- 正式使用前在一线工人配合下标定3-5个关键测温点

### ⚠️ 风险 4：WiFi 覆盖盲区

**问题**：工厂面积大，钢结构多，WiFi信号在角落可能断连。

**对策**：
- 前期实地测一下巡检路径的RSSI，确保信号强度 ≥ -70dBm
- 若存在盲区：可在外围部署一个WiFi中继（¥50）
- 断网后本地缓存数据，恢复时批量上传（MQTT离线消息可存储）

### ⚠️ 风险 5：长时间运行稳定性

**问题**：Python 开发的原型程序可能在连续运行几小时后内存泄漏或进程卡死。

**对策**：
- 使用 `watchdog` 或 `systemd` 守护进程，检测主程序无响应时自动重启
- 加入定期内存监控，超过阈值触发软重启
- 正式部署前做7×24小时压力测试
