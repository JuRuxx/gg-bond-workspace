# 技术部产线管理总则

**奥普新（陆河）科技有限公司**

---

## 一、目的与适用范围

为规范技术部与生产部在产线运行中的协作方式，建立高效、可预期的故障处理与持续改进机制，特制定本总则。

本总则适用于技术部与生产部全体人员。配套文件包括《技术部产线服务流程》（故障分级响应）与《技术部产线意见提出流程》（改进建议管理），两份流程文件为本总则的具体执行细则。

---

## 二、核心原则

### 1. 分级响应
不同问题用不同的方式处理。停线与小故障、安全隐患与操作习惯，各有对应等级和处理节奏，不搞一刀切。

### 2. 统一出口
技术部对生产部的问题对接，原则上通过值班工程师一个窗口。生产部的报修、技术部的意见，都走规定渠道，避免多头沟通造成混乱。

### 3. 对事不对人
问题描述指向具体设备、具体现象，不指向个人。理由是"设备连续报警未停机检查"，而不是"某某不会操作"。

### 4. 闭环管理
有提出必有回应，有回应必有跟踪，有跟踪必有收尾。每一个报修、每一条意见，都要走到关闭才算完。

### 5. 持续优化
不做一锤子买卖。每周有小复盘，每月有汇总，每季度有改进建议输出。把产线管得越来越好，而不是出了问题才想起来修。

---

## 三、两个流程的关系

两份流程是"修"与"改"的互补关系，共同覆盖产线管理全生命周期：

- **服务流程**（来了就修）→ 解决"现在的问题"。生产部报修，技术部响应，按P0-P3分级处理。这是**被动响应**。
- **意见流程**（修了怎么改）→ 解决"以后的问题"。技术部在服务过程中发现操作、保养等改进点，按A/B/C级提出意见。这是**主动改进**。

打个比方：服务流程是**救火**——火来了赶紧灭；意见流程是**防火**——灭了之后看看怎么不让它再着。

```
服务流程（救火）         意见流程（防火）
    ↓                         ↓
解决当下故障          →   防止同类问题复发
    ↓                         ↓
生产部发起                技术部发起
```

两条流程并行不悖，各有各的入口、等级和责任方，共同构成覆盖"修"与"改"的完整管理闭环。

## 四、角色分工

| 角色 | 职责 |
|------|------|
| **技术部主管** | 总负责本总则的推行与执行；争议仲裁（与生产部主管协商）；升级处理（内部搞不定的复杂问题）；签字确认试行期满复盘。 |
| **设备负责人** | 对应设备的专业技术支持；意见流程中A级意见的确认关卡；复杂故障的升级接收方。 |
| **值班工程师** | 日常产线报修第一响应人；B级意见的对生产部对接人；负责当日排程内的故障处理。 |
| **技术部全体** | 在服务过程中发现问题、提出意见、跟踪闭环；参与定期复盘，贡献改进思路。 |
| **生产部（协作方）** | 按规定渠道报修（P0先电话后补单，其余先填单）；接收并配合技术部提出的改进意见；安排产线人员落实整改。 |

> 角色不是岗位，是职责分工。一位同事可能同时担任多个角色，但每个角色的职责必须有人承担。

---

## 五、运作机制

### 日常运行
- 服务流程：生产部报修 → 技术部值班工程师响应 → 按P0-P3分级处理 → 异常升级机制兜底
- 意见流程：技术部发现问题 → 内部判断等级 → 按A/B/C级提出 → 跟踪直到闭环

### 定期复盘
- **每周**：内部汇总当周服务情况和意见处理进展，快速清理未闭环项
- **每月**：技术部内部集中讨论本月报修和意见数据，识别反复出现的共性问题
- **每季度**：A级+高频B级意见汇总，形成正式改进建议提交生产部主管

### 争议处理
涉及分级认定或整改责任的争议，由技术部主管与生产部主管协商裁定。双方无法达成一致的，提请双方共同上级仲裁。**任何情况下，不在产线上当场争执。**

---

## 六、附则

1. 本总则自双方主管签字之日起试行，试行期一个月。
2. 试行期满后，由技术部牵头、生产部配合，共同复盘执行效果，根据实际情况调整修订。
3. 配套流程文件（服务流程、意见流程）如有修订，以最新版本为准，本总则基本框架不变。
4. 本总则由技术部负责解释与维护。
