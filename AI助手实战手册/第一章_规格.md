## 一、文档说明：为什么要有这份规格文档

需求文档（`第一章_需求.md`）由小呆呆编写，内容全面、场景感强。但在本章落笔之前，需要先解决一个**根本矛盾**：

**需求文档写的安装流程**（Node.js → npm → npx → Claude Code CLI）  
**团队知识库查到的真实技术方案**（Claude 桌面客户端 + CC-Switch，全图形界面）

这两者的技术底座**完全不同**：

- 需求文档假设的技术栈：**终端命令行**（node -v / npx / claude-cmd）
- 确认的技术栈：**纯图形界面**（下载安装包 → 双击 → 填API Key）

目前 **尚不确定哪一套是最终方案**，因此在规格文档中同时给出两套方案的结构，并在关键节点标记 **[待确认]**。

## 五、写作规范

### 5.1 语气和人称

| 项目 | 规范 |
|------|------|
| 人称 | 统一"你"（不用"您"） |
| 语气 | "耐心的同事在教你"——不寒暄、不煽情、有温度 |
| 复杂概念的引入 | 先用比喻，再用白话解释 |
| 遇到报错 | "别慌，这个很正常"是标准开头。然后给出解决方案 |
| 遇到"你可能会问" | 加括号小字段落，比如"（你可能会问：那我打错了一个字怎么办？没事，删掉重打就行）" |

### 5.2 零基础原则

- 所有专业术语第一次出现时，必须紧跟着一句**大白话解释**（括号内或同一句）
- 不要提前出现"后面会讲"——如果某个概念本章出现但本章不解释，它就不该出现
- 每一行文字，假设读者是一个**第一次碰电脑以外复杂软件的人**

### 5.3 字数限制

| 模块 | 字数上限 | 备注 |
|------|----------|------|
| 模块一 | 300字 | 含场景描述，越短越好 |
| 模块二 | 350字 | 对照表不算字数 |
| 模块三 | 1000字 | 双系统各约500字 |
| 模块四 | 1200字 | 最长模块 |
| 模块五 | 300字 | 可以很短 |
| 模块六 | 400字 | 内容本身短，任务卡片另附 |
| 模块七 | 250字 | 越精炼越好 |
| 模块八 | 200字 | 总结 + 表格 |
| **全章** | **约4000字** | 正文，不含截图标注和表格 |

### 5.4 排版要求

- **标题层级**：`##` 模块标题 → `###` 小节 → `####` 分步操作
- **分步指南**：编号列表 `1. 2. 3.`，每步用**加粗**高亮操作动词（如"点击"、"输入"、"拖拽"）
- **截图标注**：截图下方用「图X：截图说明」的格式，加引用或斜体
- **表格**：用Markdown表格，所有表格加表头且首行加粗
- **警告框**：用 `> ⚠️ **警告**：内容` 格式高亮显示
- **提示框**：用 `> 💡 **提示**：内容` 格式
- **代码块**：推荐命令用行内代码 `` ` `` 包裹；多行命令用代码块
- **一键复制**：对学员需要输入的文字，加 `<button>复制</button>` 指示（印刷版不用，电子版需要）

### 5.5 任务卡片附录

模块六需要附加2-3个"第一次对话任务"卡片。卡片格式：

```
### 🃏 任务一：把文字改成正式邮件

> 请帮我把下面这段话改成正式的邮件格式：

**按下回车，看它怎么说～**
```

卡片独立放在章节末尾的附录中，不与正文混排。
