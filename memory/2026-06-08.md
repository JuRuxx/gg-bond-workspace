## 二、逆向工程三部曲（完整课程回顾）

### 第一课：x86 汇编基础（课程 59 分钟）

**核心知识点**：

**寄存器** — CPU 的随身笔记本
| 寄存器 | 用途 | 特殊作用 |
|:------|:-----|:---------|
| EAX | 累加器 | ★ 函数返回值放这里，逆向最常看 |
| ECX | 计数器 | for 循环的 i 存在这 |
| ESP | 栈指针 | 永远指向栈顶 |
| EBP | 基址指针 | 定位局部变量 |
| EIP | 指令指针 | 当前执行到哪一行 |
| ZF | 零标志 | CMP 后相等就置 1，JE 跳转就看它 |

**指令** — 看懂 80% 反汇编代码只要 10 条指令
- MOV：搬数据（类比 PLC 的 MOV）
- PUSH/POP：压栈/出栈（LIFO，像产线的栈式料架）
- CALL/RET：函数调用（调用→执行→返回）
- CMP/JMP/Jcc：比较和跳转（if/else 翻译成汇编就是 CMP+JE/JNE）
- XOR：异或，常见清零操作
- LEA：取地址（相当于 & 运算符）

### 第二课：PE 结构解析（课程 56 分钟）

**核心知识点**：

**PE 文件 = Windows 上的 .exe / .dll / .sys**

```
PE 文件三明治结构：
┌─ DOS MZ 头（64字节）──┐  ← e_lfanew 指向 NT 头
┌─ NT 头 ────────────────┐  ← Signature("PE\0\0"), FileHeader, OptionalHeader
│  ┌ FileHeader         │     Machine(CPU类型), NumberOfSections, Characteristics
│  └ OptionalHeader     │     ★ AddressOfEntryPoint(入口点) ★ 数据目录表
├─ 节区表 ───────────────┤  ← 节区索引
├─ .text (代码段) ──────┤  ← 可执行，代码在这
├─ .data (数据段) ──────┤  ← 全局变量
├─ .rdata (只读数据) ───┤  ← 字符串、导入表
├─ .idata (导入表) ─────┤  ← ★ 调用了哪些 DLL 函数
├─ .rsrc (资源) ────────┤  ← 图标、字符串表
└─ .reloc (重定位) ─────┘  ← DLL 用
```

**最关键的三个字段**：
1. **Machine**（0x14C=32位, 0x8664=64位）→ 选对调试器
2. **NumberOfSections** → 正常 4-6 个，2 个说明被加壳了
3. **Characteristics** → 判断是 EXE 还是 DLL

**导入表 = 程序的电话簿**，看它就知道：
- 用了哪些外部函数（有 `CreateFileA`+`DeviceIoControl`→ 跟硬件通信）
- 有没有加密狗（`sentinel.dll` / `hasp_api.dll`）
- 有没有加壳（只有 `LoadLibraryA`+`GetProcAddress`+`VirtualProtect`→ 加壳了）

### 第三课：反调试技术（课程 30 分钟）

**核心知识点**：

反调试三板斧：
1. **检测调试器** — `IsDebuggerPresent()`、`NtGlobalFlag`、PEB 检测
2. **干扰调试器** — 代码自修改、垃圾指令
3. **时间炸弹** — `rdtsc` 时间差检测

**五种常见手段及绕过方式**：
| 手段 | 原理 | 绕过方法 |
|:----|:-----|:---------|
| IsDebuggerPresent | 读 PEB.BeingDebugged | 改返回值或改跳转 |
| NtGlobalFlag | 读 PEB 的调试标志 | 同上，或用 ScyllaHide |
| rdtsc 时间差 | 测量代码执行时间，单步太慢 | 硬改 EAX 寄存器 |
| TLS 回调函数 | 入口点之前执行反调试 | x64dbg 设置"在 TLS 回调中断" |
| 代码自修改 | 运行时改自己代码 | 等修改完再分析，或用硬件断点 |

## 三、主人提问详细解答

### 问题①：PC 结构 中的那三个字段 为什么逆向关心
*（提问点：Machine、NumberOfSections、Characteristics）*

**展开解释**：

这三个字段是打开一个 PE 文件后**第一个**看到的信息，直接决定后续操作。

#### Machine — 判断 32 位还是 64 位

**重要性**：用错调试器会直接报错，白忙活。

```
0x14C (x86 32位)   → x32dbg
0x8664 (x64)       → x64dbg
```

**更深层的影响**——32 位和 64 位的代码完全不同：

```asm
; 32 位 (x86) 反汇编
PUSH EBP           ; 参数压栈传递
MOV EBP, ESP

; 64 位 (x64) 反汇编
PUSH RBP           ; 寄存器变 R 前缀
MOV RBP, RSP
; 前4个参数: RCX, RDX, R8, R9（寄存器传参，不压栈！）
; 第5个参数开始才压栈
```

**地址长度也不一样**：
- 32位：地址 4 字节，范围 0x00000000–0xFFFFFFFF
- 64位：地址 8 字节，范围 0x0000000000000000–0xFFFFFFFFFFFFFFFF

**如果你用 x32dbg 打开 64 位程序**：
```
x32dbg 报错："不是有效的 32 位 PE 文件"
→ 你浪费了 30 秒（还好，但如果是在分析过程中才发现就很烦）
```

**实践建议**：拿到一个 exe，第一件事不从双击开始——先扔进 PE-bear 看 Machine。

#### NumberOfSections — 判断有没有加壳

**重要性**：加壳的程序直接反汇编会看到乱码，要先脱壳。

```
NumberOfSections = 6 = 正常 VC++ 编译的程序
NumberOfSections = 2 = 被 UPX 压缩的程序

为什么被压缩只有 2 个节区？
  UPX 把代码、数据全部打成压缩包，只留：
  UPX0 = 壳的解压代码
  UPX1 = 原程序的压缩数据
```

**节区名也暴露一切**：
```
.text → 正常
CODE  → Delphi 编译
UPX0 UPX1 → UPX 加壳
.aspack → ASPack 加壳
.themida → Themida 强壳（很难脱）
vmp0 vmp1 → VMProtect（虚拟化保护，非常难分析）
```

**权限异常也是线索**：
```
节区名: .text
权限: 可读 + 可写 + 可执行（0xE0000020）
→ 极其可疑！正常 .text 不可写
→ 说明代码会在运行时被修改（加壳或恶意软件）
```

#### Characteristics — 判断是 EXE 还是 DLL

**重要性**：DLL 不能直接双击运行，调试方式也不同。

```c
// Characteristics 是一个 16 位的位图
if (Characteristics & 0x2000) {  // IMAGE_FILE_DLL
    printf("这是 DLL，需要注入到进程才能调试");
} else {
    printf("这是 EXE，双击就能跑");
}
```

**调试 DLL 和 EXE 的区别**：

```
DLL 调试流程：
1. 打开 x64dbg → 文件 → 附加（Attach）
2. 选择正在运行的目标进程（比如 notepad.exe）
3. 让目标进程加载这个 DLL
4. 在 DllMain 入口断下来

EXE 调试流程：
1. 打开 x64dbg → 文件 → 打开（Open）
2. 直接加载 .exe
3. 断在入口点 AddressOfEntryPoint
```

#### 为什么逆向者看这三个字段？

```
拿到一个 exe → 3 秒内做出判断：

Machine = 0x14C → 32位 → 开 x32dbg
Machine = 0x8664 → 64位 → 开 x64dbg

NumberOfSections = 6 → 正常，直接反汇编
NumberOfSections = 2 → UPX0/UPX1 → 先脱壳

Characteristics & 0x2000 = 真 → 这是 DLL，要附加到进程
Characteristics & 0x2000 = 假 → 这是 EXE，双击就能跑
```

### 问题②：导入表（Import Table）

**展开解释**：

**导入表 = 程序的电话簿**。程序自己不写所有功能，而是调用 Windows 现成的 API。导入表记录了：**"我要调哪些 DLL 的哪些函数"**。

```c
typedef struct _IMAGE_IMPORT_DESCRIPTOR {
    DWORD OriginalFirstThunk;   // → 指向函数名数组（原始备份）
    DWORD TimeDateStamp;
    DWORD ForwarderChain;
    DWORD Name;                  // → 指向 DLL 名字符串
    DWORD FirstThunk;            // → 指向 IAT（导入地址表，运行时填地址）
} IMAGE_IMPORT_DESCRIPTOR;
```

**为什么有 `OriginalFirstThunk` 和 `FirstThunk` 两个指针指向同样的内容？**

```
文件中：
  OriginalFirstThunk → [函数名1, 函数名2, ...]  ← 永久保留
  FirstThunk → [0, 0, 0, ...]                    ← 占位

运行时（Windows 填完地址后）：
  OriginalFirstThunk → [函数名1, 函数名2, ...]  ← 还在
  FirstThunk → [0x77665544, 0x7766AABB, ...]     ← 已填入实际内存地址
```

**逆向者用导入表做什么？**

```c
// 例1：判断是否用加密狗
如果导入表里有：
  sentinel.dll / hasp_api.dll / hardlock.dll
  → 有加密狗！

// 例2：判断是否跟硬件设备通信
kernel32.dll:
  → CreateFileA     // 打开设备
  → DeviceIoControl // 向设备发命令
  → CloseHandle     // 关闭设备
  → 跟硬件通信！

// 例3：判断是否被加壳
如果导入表只有三条：
  → LoadLibraryA     // 加载 DLL
  → GetProcAddress   // 获取函数地址
  → VirtualProtect   // 修改内存权限
  → 加壳了！真正的导入表被压缩了
```

### 问题③：节区（Section）

**展开解释**：

**节区 = PE 文件里的"文件夹"**，把不同种类的内容分开放，因为需要不同的访问权限。

```
文件里（硬盘上）：
┌─ DOS头 ─┐
├─ NT头 ──┤
├─ 节区表 ─┤           ← 相当于"索引目录"
├─ .text ─┤           ← 这个文件夹放代码（权限：可读+可执行）
├─ .data ─┤           ← 这个文件夹放变量（权限：可读+可写）
├─ .rdata ─┤          ← 这个文件夹放字符串（权限：只读）
└─ .rsrc ─┘           ← 这个文件夹放资源（权限：只读）
```

**每个节区对应一个 `IMAGE_SECTION_HEADER`（40字节）。** 关键的四个字段：

```c
typedef struct _IMAGE_SECTION_HEADER {
    BYTE  Name[8];          // 名字（.text / .data / .rdata）
    DWORD VirtualSize;      // 内存中占多大
    DWORD VirtualAddress;   // 内存中起始地址（RVA）
    DWORD SizeOfRawData;    // 文件中占多大
    DWORD PointerToRawData; // 文件中起始位置（偏移）
    DWORD Characteristics;  // 权限标志（可读/可写/可执行）
} IMAGE_SECTION_HEADER;
```

**为什么要有节区？**

因为不同类型的内容需要不同的内存访问权限：

| 节区 | 需要什么权限 | 为什么 |
|:----|:------------|:------|
| .text（代码） | 可读 + **可执行** | CPU 要读指令来执行 |
| .text（代码） | **不可写** | 防止代码被改 |
| .data（变量） | 可读 + **可写** | 程序要修改变量值 |
| .data（变量） | **不可执行** | 防止数据被当成代码执行 |
| .rdata（只读） | **只读** | 字符串不能改 |

**NumberOfSections 的值怎么看？**

```
4–6 → 正常程序
2   → 被 UPX 压缩（节区名是 UPX0/UPX1）
```

**节区名也有猫腻**：不同编译器起的名字不一样，有些恶意软件会改节区名伪装。
